简单总结mmdetection源码TwoStageDetector的流程细节(五) – 解析BBox head forward 以及相关 Loss

一、前言

  1. 上一篇总结介绍了关于 GT assign 以及 Proposal Sample 的整体流程, 这个部分就是 TwoStageDetector.forward_train 的最后的部分了(不考虑 mask 相关部分), 但是到这里还是有疑问没有解决的, 具体疑问看 3.
  2. 本篇总结将会对 BBox head forward 以及相关 Loss 部分(mmdetection代码的注释是这么写的)进行解析.
  3. 疑问就是从上一个总结就有的, 关于多尺度信息在训练的时候怎么去区分, 因为上一篇文章也注意到了, 进行 GT BBox 匹配的时候也并没有区分什么尺度问题, 都是根据 IoU 在进行的计算, 然后进行筛选, 仅由的针对多阶段的过程还是 RPN 部分的 rpn_head.get_bboxes 其将多尺度的信息直接并成了一个维度, 只是有一个选项是选择是否进行跨尺度的 nms 操作. 希望在本文的 roi_extractor 部分会涉及到如何进行尺度分配的部分
    • 问题找到答案了: 在 mmdet.models.roi_extractors.single_level.py 中的 SingleRoIExtractor 类的 forward 过程中 会进行 map_roi_levels 函数的处理, 会根据 RoI 的大小去分配对应的 stage
  4. 另外的一个疑问: 从这部分的代码中已经可以看出了, 提取出来的 features 是来自于各种不同 scale 的啊, 后续的分支是公用的吗???

二、总体流程

  1. 首先将上一个阶段得到的 sampling_results 中的每一个 img 的 SamplingResult 对象提取出来其中的具体信息( torch.cat([self.pos_bboxes, self.neg_bboxes]) ), 然后进行 bbox2roi 操作, 就是添加了一下所属图片的信息
  2. 重点: 使用 bbox_roi_extractor 进行每个 roi 的特征信息的提取
  3. 然后将得到的 features feed 给 bbox_head (这里包含了 classification 以及 bbox regression)
  4. 最后将 loss 记录一下就可以了
  5. 然后整个 train_forward 部分就结束了, 将 losses 返回就好了 (nice!!!)

    • (2). 看看 forward 部分:

      • 1). 首先如果 feats (也就是各个 stage 的 feature map) channel 数目只有 1(就是不使用 FPN 的 Faster RCNN) 的时候, 就直接根据仅有的 RoILayer 进行处理然后返回即可 即 return self.roi_layers[0](feats[0], rois), 但是如果使用 FPN (即 channel 数目 > 1)那么就要进行下面的操作了
      • 2). 首先会获取一些维度信息, 然后使用 map_roi_levels 对 RoI 进行 stage 的分配 (就是这里, 就是这里, 这里将不同的 RoI 按照大小分配到不同的 stage 上), 康一康 map_roi_levels:
        • map_roi_levels(self, rois, num_levels): rois 就是所有的 RoI 框. num_levels: 使用的 feautre map stage 的数目
        • 具体操作就是: 先算一下每个 proposal 的面积, 然后除以 finest_scale 之后进行一下 log2, 取一下floor, 之后 clamp 到 0 - 3(这个 config 来看是 3) 之间. 然后返回就可以

      • 5). 返回这个 roi_feats

    • 也就是我们首先会经过 classfication 与 bbox regression 共同要使用的 conv, fc 层, 然后才拆分为两个分支, 分别进行两个任务的 conv, fc 层, 最后得到结果
    • 所以有了这个目的, 让我们看一下其中的参数吧, 但是还是最好先看一下其父类 BBoxHead 的参数:
    • (2). BBoxHead.__init__(self, with_avg_pool=False, with_cls=True, with_reg=True, roi_feat_size=7, in_channels=256, num_classes=81, target_means=[0., 0., 0., 0.], target_stds=[0.1, 0.1, 0.2, 0.2], reg_class_agnostic=False, loss_cls=dict( type='CrossEntropyLoss', use_sigmoid=False, loss_weight=1.0), loss_bbox=dict( type='SmoothL1Loss', beta=1.0, loss_weight=1.0)):

    • (3). ConvFCBBoxHead.__init__(self, num_shared_convs=0, num_shared_fcs=0, num_cls_convs=0, num_cls_fcs=0, num_reg_convs=0, num_reg_fcs=0, conv_out_channels=256, fc_out_channels=1024, conv_cfg=None, norm_cfg=None, *args, **kwargs):

      • 处理之后得到 labels, label_weights, bbox_targets, bbox_weights 然后会返回给 bbox_target, 然后在其中经过汇总拼接将拼接好的多尺度的信息返回
  5. 我们已经拿到了 optimize target 了, 接下来就要去拿 loss 了, 通过 bbox_head.loss 按照配置文件进行 loss 的计算配置 然后返回 loss
  6. 我们拿到了 loss 之后, 在 two_stage.py 的 forward_train 中返回这些 loss 即可

二、该部分的总体流程

PS: 虽然在上一个阶段 RPN 相关部分就已经涉及到了 assigner 以及 sampler 的使用, 但是没有进行分析, 给忽略掉了… (在 anchor_target 部分中使用的), 本文尽量对这两个东西进行一下分析, RPN 以及 RCNN 中使用的 assigner 以及 sampler 的配置除了参数设置稍微不同之外都是相同的.

  1. 根据 cfg.train_cfg.rcnn.assigner 进行 bbox_assigner 的构建
  2. 进行 cfg.train_cfg.rcnn.sampler 进行 bbox_sampler 的构建
  3. 对 batch 中的每个 image 使用 bbox_assigner 进行 assign 操作
  4. 继续对 assign 操作得到的结果使用 bbox_sampler 进行 sample 操作 得到这个阶段的最后的结果

PS: assigner 以及 sampler 的部分分别在: mmdet.core.bbox.assigner 以及 mmdet.core.bbox.sampler 两个 module 中

PS: 该部分正常的整体思路描述

  1. 每个 proposal 先赋值为 -1 (-1 表示不会care这些anchor)
  2. 某个候选框和所有 GT bbox 的 IoU 阈值都小于 negative IoU 阈值, 赋值为 0
  3. 对某个候选框, 如果与其有最大阈值的 GT bbox 的 IoU 阈值大于 positive IoU 的时候, 会将这个 proposal 的值赋值为这个 GT bbox 的编号
  4. 因为前三步是对每个 proposal 进行与 GT bbox 的匹配以及进行编号安排, 所以仍然可能存在 GT bbox 没有被匹配到, 所以要对 所有 GT bbox 再进行一次判断, 与其 IoU 最大的 proposal 将会被赋值为这个 GT bbox 的编号.
  5. 匹配完毕之后进行采样, 会按照一定的比例进行采样(due to the positive-negative imbalance problem)

三、各步骤细节

PS: 无论是 assign 还是 sample 都是分别针对单张图片进行的

  1. 首先对 Faster-RCNN 中使用的 assigner – MaxIoUAssigner 进行分析
    • PS: 首先看到了其有父类 BaseAssigner, 但是看到了父类只是一个虚类, 并且只有一个并没有实现的抽象方法, 所以不用看 BaseAssigner 了
    • (1). 首先还是看 MaxIoUAssigner 的 __init__ 函数:
    • __init__(self, pos_iou_thr, neg_iou_thr, min_pos_iou=.0, gt_max_assign_all=True, ignore_iof_thr=-1, ignore_wrt_candidates=True):
      • 1). pos_iou_thr: 看做正样本所需要的 IoU 大小阈值
      • 2). neg_iou_thr: 小于这个值的被看做为负样本
      • 3). min_pos_iou: 看做正样本所需的最小的 IoU 大小阈值
      • 4). gt_max_assign_all: 对于一个 GT 有多个 bbox 都与其有最高的 IoU 的时候, 是否将这些 bbox 都 assign 给这个 GT bbox.
      • 5). ignore_iof_thr: 忽略 GT bbox 的 iof 阈值(???什么阈值???), 当取负值的时候不会忽略任何 GT bbox
      • 6). ignore_wrt_candidates: 是否去计算 bboxes 与 gt_bboxes_ignore 之间的 iof
      • __init__ 函数中就是简简单单的进行 这些参数在类内的赋值
    • (2). assign 函数: assign(self, bboxes, gt_bboxes, gt_bboxes_ignore=None, gt_labels=None):
      • 1). bboxes: 就是所有在上一阶段整合处理之后的 proposals, (n, 4)的 Tensor
      • 2). gt_bboxes: 就是 GT bboxes信息, (k, 4)的 Tensor
      • 3). gt_bboxes_ignore: 忽略掉的 gt bbox
      • 4). gt_labels: GT bboxes的 Label
      • 首先会通过 bbox_overlaps 函数计算 overlaps
      • 然后会针对是否忽略框进行判断, 如果忽略的话进行一些操作(PS: 主要是筛选掉一些overlap项, 因为暂时用不到, 先跳过 233)
      • 之后就会使用得到的 overlaps 去进行 assign_wrt_overlaps 操作, 然后将得到的结果进行返回, 这部分结束, 继续看 assign_wrt_overlaps 函数…
    • (3). assign_wrt_overlaps 函数: assign_wrt_overlaps(self, overlaps, gt_labels=None)
      • PS: 这个就是真的按照上面的流程说的处理的了
      • 1). 首先就是使用 torch.Tensor.new_full 创建了一个 size 为 (num_bboxes, ) 的用 -1 填充的 Tensor, 其 dtype 为 torch.long
      • 2). 对每个 proposal 获取到有最大 overlap 的 GT IoU 值以及对应的 index -> max_overlaps, argmax_overlaps 分别代表着 最大的 overlap 值以及 index
      • 3). 对每个 GT bbox 获取到有最大 overlap 的 GT IoU 值以及对应的 index -> gt_max_overlaps, gt_argmax_overlaps
      • 4). 利用上面的到的参数 max_overlaps 对应 index 处的值如果小于 neg Iou threshold 那么就会赋值为 0 (PS: neg IoU threshold 也可以设置为一个 tuple 代表阈值在这个范围内的视为 negative sample)
      • 5). 进行 positive sample assign, 对所有 proposal 最大 overlap 的 GT IoU 大于 pos IoU threshold 的赋值为 对应的 GT bbox 编号
      • 6). 然后会对所有的 GT bbox 去找最大的 IoU bbox, 这个 IoU 如果没有超过 pos IoU threshold 的时候就放弃, 超过了就根据设置如果对所有相同 IoU 的都进行 assign 就对所有的对应的进行当前 GT bbox 编号的 assign, 否则就针对一个进行 (PS: 按照 torch.max 的逻辑, 这个通常都是最后一个…)
      • 7). 当传入的参数 gt_labels 非空的时候, 在这里就会 添加一个 assigned_labels 对应于该 proposal 对应的 GT bbox 编号类别 label.
      • 8). 最后返回的值会封装在 AssignResult 中
      • PS: em… 7)…
  2. 接下来对使用的 RandomSampler 进行分析:
    • (1). 首先对 __init__ 函数进行分析: 其会调用父类 BaseSampler 中的 __init__ 函数: __init__(self,num,pos_fraction,neg_pos_ub=-1,add_gt_as_proposals=True,**kwargs)
      • 1). num: 提取正负样本数量
      • 2). pos_fraction: 正样本比例
      • 3). neg_pos_ub: 最大负样本比例, 大于该比例的负样本会被忽略, -1 表示不忽略
      • 4). add_gt_as_proposals: 是否将 GT 加入 proposal 作为正样本
      • 操作: 只是简单的将参数进行赋值而已
    • (2). 那么接下来就对 forward_train 中使用的 sample 函数进行分析就可以了: 仍然是在父类 BaseSample 中实现的
      • sample(self, assign_result, bboxes, gt_bboxes, gt_labels=None, **kwargs):
      • 1). assign_result: 上一步进行 proposal 与 GT BBox assign 的结果
      • 2). bboxes: rpn forward 之后的 proposal 结果
      • 3). gt_bboxes: GT bboxes 信息
      • 4). gt_labels: GT labels 信息
      • 流程:
      • 1). 首先还是会从 proposal 中提取前四个维度(也就是 bbox 的信息, 不包含 score)
      • 2). 创建了一个 gt_flags Tensor, 暂时将所有的 bbox 对应位置都设置为 0 (表示不是 GT)
      • 3). 如果选择将 GT bbox 作为 proposal (即 add_gt_as_proposals == True), 那么会将 GT BBox cat进入, 并且会在 assigner 中加入这些, 然后将这些的 gt_flags 设置为 1 (表示是 GT) 拼接进去
      • 4). 使用 _sample_pos sample 出按照设置中数量的 positive samples, 代码中提到可能会有重复的 index 被 sample 出来(代码认为可能是 pytorch 的 bug), 所以要进行去重复的操作, 然后剩下的 samples 数量都是暂时需要的 negative samples 的数量了
      • 5). 如果在 config 中设置的 neg_pos_ub 非负, 按照这个比例设置 negative samples 数量的上限
      • 6). 使用 _sample_neg sample negative samples, 并且进行去重
      • 7). 将 pos_ind, neg_ind, bbox, gt_bboxes, assign_result, gt_flag 使用 SamplingResult 进行封装, 然后返回
  3. 最后我们将每个图片得到的 SamplingResult 实例放入 sampling_results 中, 这个阶段就暂时结束了.

  1. 首先看一下 RPNHead.__init__ 函数: __init__(self, in_channels, **kwargs):

      • (1). num_classes: 分类数. Faster-RCNN只是简单的分为前景和背景, 所以 num_classes 赋值为 2. (PS: 如果是 one-stage 的方法, 这里的类别通常就是81了)
      • (2). in_channels: 输入进来时候的通道数. 之前在 FPN 类的总结里已经看过源码了, 每个层次都会被卷积转化为 256 channel, 所以 Faster-RCNN 的 config 文件中这个值取 256
      • (3). feat_channel: (默认值为256), 表示特征层的通道数(就是根据 FPN 处理之后的 stage output 后继续使用一个 conv 进行处理, 这个 conv 处理之后 output 的 channel 数量), config 中是 256

      • (6). anchor_strides: (默认值为[4, 8, 16, 32, 64]), 不同尺度 feature map 上每一个位置处, 一个 pixel 对应的 stride 大小.
      • (7). anchor_base_sizes: (默认值是None), 代表一个 pixel 占据的长宽大小是多少 值等于 sqrt(w*h) (PS: 这个值当取 None 的时候, 将会与 anchor_strides 相同)
      • (8). target_means: (默认值是(.0, .0, .0, .0))
      • (9). target_stds: (默认值是(1.0, 1.0, 1.0, 1.0))
      • (10). loss_cls: (默认值看上面吧), 表示 RPN 部分的 classification loss
      • (11). loss_bbox: (默认值看上面), 表示 RPN 部分的 bbox regression loss

    • (2). 每一个 stage 的 anchor 的数量为 self.num_anchors = len(self.anchor_ratios) * len(self.anchor_scales)
    • (3). 最后会构建一下 RPN 阶段的 网络结构的卷积层, 使用 _init_layers 函数, 会构建三个 conv 层(RPNHead类中重写的是3个, AnchorHead类中是2个), 一个 rpn_cls, 一个 rpn_reg, 一个 rpn_conv. 具体信息看代码吧.

    • PS: forward 中的 multi_apply 还是要看一下的… 其是为了处理多尺度的, 因为 fpn 输出的是多个 stage 的, 需要每个 stage 进行单独的 forward_single 函数的调用

    • (7). 如果设置中设置了要进行跨尺度的 nms, 那么进行跨尺度的 nms

    • (9). 最后将筛选出的 proposals 返回, 这就是一个 image 对应的 proposals 了, 多个 image 将会最后再 放入另外的一个 list 中, 然后返回, 我们就得到了 proposals 列表了

    • (5). end_level: 默认值是-1
    • (6). add_extra_convs: 默认值是False. 表示是否使用 C6 层进行detection
    • (7). extra_convs_on_inputs: 默认值是True. PS: 没有看到使用
    • (8). relu_before_extra_convs: 默认值是False PS: 没有看到使用
    • (9). no_norm_on_lateral: 默认值是False. 是否在 lateral_conv 中使用 norm. False 的时候 lateral_conv 构建的时候 norm 就是 None. True 的时候会采用和 norm_cfg 相同的值
    • (10). conv_cfg: 默认值是None. 进行 conv 构建的时候的参数, 在 conv_module.py 中可以发现, 这个值好像就是用来选择 conv 的类型的. 默认使用的是 torchnn.Conv2d.
    • (11). norm_cfg: 默认值是None. 进行 conv_module 构建中的 norm 层构建的 cfg. 在 norm.py 中可以知道, 这个参数是用来选择 norm 层的类型的. 默认下(None)不会添加 norm 层.
    • (12). activation: 默认值是None
    • 开始进行 __init__ 函数的流程分析:
    • (1). 首先进行了各种class内值的赋值, 新增了 self.num_ins = len(in_channels) 参数, 其代表了从Backbone中提取阶段的数量. 然后还进行了一个判断: 当 end_level 为 -1 的时候, self.backbone_end_level = self.num_ins, 此时 backbone 结束阶段就是从 backbone 中提取阶段的总数, 否则 backbone 中提取的阶段总数 self.backbone_end_level 等于 end_level 的大小
    • (2). 从 start_level 的值开始到 backbone_end_level 的值结束, 进行 lateral_convs 以及 fpn_convs 的构建. 两种 conv 层的区别是: lateral_conv 是 1 1 尺寸的卷积核, norm 由 no_norm_on_lateral 进行调控, 没有 padding, in_channels 和 out_channels 大小不同, 就是由 __init__ 的输入的参数所决定的([256, 512, 1024, 2048], 256). 而 fpn_convs 是 3 3 尺寸的卷积核, padding 为 1, in_channels 和 out_channels 的大小都是 256.
      • 由此可以看出 fpn_convs 是 lateral_convs 处理 backbone 的各个 ouput stage 之后进一步进行处理的 conv 层.
    • (3). 会根据 add_extra_convs 参数以及 extra_levels = num_outs - self.backbone_end_level + self.start_level 决定是否添加额外的 conv 层. (PS: 代码里写了, 比如: RetinaNet(用了C6层)). 因此虽然按照 Faster-RCNN 的配置来说 extra_levels = 5 - 4 + 0 = 1, 但是 add_extra_convs 的值是 False, 所以并不会额外的添加 conv 层… 哦豁… 然而后面的 forward 可以发现, 虽然没有添加 conv 层, 但是 num_outs 参数还是会决定输出多少个 stage 的 output
  2. init_weights 就是进行 self.modules() 得到的所有 Conv 层进行初始化
  3. forward 函数:
    • PS: forward 函数是使用了 @auto_fp16() 装饰器的… 是强制使用 fp16 进行训练的, 整个网络本质上是混合 fp16 以及 fp32 进行训练的. 好像 resnet 部分就没有强制使用 fp16 训练.
    • (1). 首先 forward 函数只有一个参数: inputs, 这个 inputs 就是从 backbone 中得到的一组 output. 其channel 数量分别对应着 in_channels 各个 entry 的值的大小.
    • (2). 首先使用各个 stage 的 output 用 lateral_conv 进行处理 得到一组输出 laterals
    • (3). 按照 fpn 的上下层融合的思想 利用后一层的 upsample(这里使用的是nearest) 和当前层次进行进行加法, 对 laterals 进行更新
    • (4). 然后使用 fpn_convs 进行得到的 laterals 的 conv 处理, 得到了 outputs
    • (5). 接下来会进行 extra stage 数据的产生了, 对于 Faster-RCNN 等网络来说, add_extra_convs 是False, 只是会根据 num_outs 参数通过 max_pool 补上后面的 stage 的 output, 但是对于其他的比如 RetinaNet 等网络来说, 有 extra_convs 的话会进一步根据情况进行后续的 通过 conv 得到 stage output.
    • (6). 最后会将各个 stage 的 outputs -> outs 用 tuple 封装然后返回.

PS: 到此为止 FPN 类的解析就完全结束了

(二)、加载数据集:

mmdet/datasets/builder.py build_dataset函数:

  1. 传入的参数为: cfg(dict类型), default_args(暂时不知道这个是什么用, 一直传递但是一直没有怎么用过, 可能是为了以后扩展方便???)
  2. 通常来说传入的cfg就是一个数据库配置的参数字典, 但是其还是有其他的选择的, 几种选择如下所示:
    • (1). 传入一个为 dict 类型的 data cfg, 且其中的anno_file只涉及到一个数据集的名字: 会正常的使用 build_from_cfg 函数(来自 utils/registry.py 文件) 去构建数据集
    • (2). 传入的是 tuple或者list 类型的时候: 会将其一个个的拆分, 每个使用 build_from_cfg 函数去进行构建, 然后将这些得到的数据集, 使用ConcatDataset类进行封装处理
    • (3). 传入的是 dict 类型的data cfg, 但是anno_file涉及到了多个数据集的名字, 会使用 _concat_dataset 函数进行处理来得到最终的数据集
    • (4). 传入的是 dict 类型的 data cfg, 但是 cfg 中 type 参数为 RepeatDataset, 会使用 RepeatDataset 进行 cfg times 参数的次数的重复的组合

mmdet/datasets/loader/build_loader.py build_dataloader函数:

  1. 函数定义: build_dataloader(dataset, imgs_per_gpu, workers_per_gpu, num_gpus=1, dist=True, **kwargs):
    • (1). dataset就是之前用build_dataset构建的Dataset相关信息, 但是只是存储了anno_file等文件的路径作为参数, 并没有真正的将数据加载进来.
    • (2). imgs_per_gpu: 每个GPU分配的image数量
    • (3). workers_per_gpu: 每个GPU分配的提取图片用的线程
    • (4). num_gpus: 使用gpu的数量:
    • (5). dist: distribution缩写, 表示是否使用分布式
    • (6). **kwargs: 所有的这部分参数后面会传递进入Dataloader中, shuffle指的是是否打乱.
  2. PS: 针对分布式的先不管, 目前就针对单机进行分析
  3. batch_size = imgs_per_gpu num_gpus num_workers = workers_per_gpu num_gpus
  4. 使用的GroupSampler是基于pytorch的Sampler基础上实现的, 用来控制数据后续的sample. PS:其中调用了一个Dataset的参数flag: 看注释说flag是进行图片分组的, 图片aspect ratio > 1的flag为1, 否则flag为0
  5. GroupSampler初始化的整个流程
    • (1). self.group_size = np.bincount(self.flag) 是对两组(根据aspect ratio划分)的计数
    • (2). self.num_samples是两部分count分别除以samples_per_gpu(就是imgs_per_gpu)的向上取整再乘以samples_per_gpu, 只是为了保证数量是可以被整除的
  6. GroupSampler中的重写Sampler的函数:
    • (1). len(self): 就是简单的返回self.num_samples
    • (2). iter(self): 会对两个group进行分别的操作, 因为上面说了会为了保证每个组中的元素是被samples_per_gpu所整除的, 所以在使用的时候对对应的group的index数组进行shuffle之后 会用前面的部分来填补到想要保证的length
  7. PS: GroupSampler相当于只是对图片进行了随机的打乱, 方便后续的sample的随机性
  8. dataloader使用的就是torch提供的dataloader, 只不过sampler以及collate_fn是框架内实现的.
    • (1). sampler: 这个sampler就是刚才使用的GroupSampler这个东西。
    • (2). collate_fn: 这里是为了扩展到并行计算的时候使用的自定义的collate, 暂时略过, em… 是进行数据分配的
    • PS: pin_memory这个东西是选是否使用锁页内存, 在高端机器, 高内存剩余的时候选择设置为True即可

(三)、从配置文件中进行模块的构建

mmdet/utils/registry.py文件 中的 build_from_cfg 函数:

  1. build_from_cfg(cfg, registry, default_args=None): cfg是config dict, registry
  2. 之后还是会向要构建的Module传入cfg作为参数, 所以这个就需要config的参数和不同部分组成模块的统一设置
  3. PS: 本质意义其实是, 在test开始的时候其实已经引入了很多的类, 包含了各种各样的模型类可以进行调用, 然后使用这个函数去根据conf, 将类进行实例化出来

mmdet/utils/registry.py文件 中的 Registry 类:

  1. 其包含了两个参数: name以及module_dict. 前者是指定了该Registry类别的, 后者是一个 dict, 里面包含了该Registry中的module信息. 这个 dict 内部是 module_name: class 的形式
  2. 实际上name就是其类别, module_dict 是type: type对应class
  3. PS: Registry类框架中抽象出来了一些预先设置好的Registry, 然后后续会用Registry中的方法register_module做修饰器, 去使得各个模块一开始就加载进来了, 而不是后续去进行模块的创建.
  4. 重点: 刚开始看代码的时候, 发现在使用build_from_cfg函数进行dataset构建的时候, build_from_cfg要求Registry对象中必须要包含 cfg 中 type 名称的对应 class. 但是代码中构建数据集的时候传递的参数是 DATASETS, 其只是简单的设置了一个dataset作为Registry对象的name, 而module_dist中是没有任何赋值的. 所以按理来说是要报错的. 但是框架的处理方法是: 使用@DATASETS.register_module作为修饰器放置在每个数据库类型的Dataset Class上, 同时在package的init.py中将这些Class全部引入一遍, 此时 DATASETS 就已经注册了所有的Datasets Class(在 import 那个模块的任意东西的时候就已经导入了) 所以实际上就已经提前的引入好了

PS: mmdetection框架将整个network分成了几个部分: mmdetection的github doc中说大致上分成了四个部分: 1. backbone: 就是网络前面的提取特征的backbone, eg: ResNet 2. neck: 在backbone与head中间的部分, eg: FPN, HRFPN(HRNet提出的), BFP(Libra R-CNN的那个), ASPP 3. head: 针对特定任务的部分, eg: bbox prediction, mask prediction 4. roi extractor: 从feature map中提取feature 的部分, eg: RoI align.

    • (1). backbone: 网络的backbone部分的config dict
    • (2). neck: 网络的neck部分的config dict
    • (3). shared_head: shared_head部分的config dict(PS: 在shared_head module里只看到了一个ResLayer…)
    • 后面同理, 直到mask_head都是网络对应部分你的config dict
    • train_cfg, test_cfg仍然是训练时候和测试时候的config dict, pretrained是是否有预训练模型
  2. 之后在TwoStageDetector.init()中就都是使用 mmdet/detectors/builder.py 中的各个build函数进行各个部分的根据build_from_cfg的构建的(PS: 每个部分仍然是都有自己单独的Registry, 且在其部分module的init文件中已经被注册进入). 最后会进行pre_train weight的加载(只在train过程中有可能进行了加载, 在inference中因为test.py中设置了cfg中的model.pretrain=None, 所以是不会进行加载的, 而是在model建立之后才进行checkpoint的加载)
  3. 关于pre_train weight的加载, 除了backbone以及shared_head部分的初始化是使用pre_train文件中的weight了, 其余部分的初始化都是根据各个module中自身定义的初始化函数去进行定义的, 大部分使用的都是pytorch自带的normal_init进行的参数的初始化
  4. 其他函数:
    • (1). init_weights(self, pretrained=None): 就是使用pre_train参数进行model初始化的函数
    • (2). extract_feat(self, img): 从backbone+neck中进行feature的提取, 其实就是将input经过backbone以及neck部分的forward而已
    • (3). forward_dummy(self, img): 计算网络的flops
    • (4). simple_test(self, img, img_meta, proposals=None, rescale=False): 只使用单一尺度的tesst
    • (5). aug_test(self, imgs, img_metas, rescale=False): 使用数据增强, multi-scale的test
    • PS: 测试过程会在BaseDetector根据img的第一维度长度进行 aug_test还是simple_test的选择.
    • (6). forward_train(self, img, img_meta, gt_bboxes, gt_labels, gt_bboxes_ignore=None, gt_masks=None, proposals=None): 训练过程中向前传递的过程, 会返回loss, 具体部分后面分析
    • PS: (4). (5). (6). 三个函数实际上是在 BaseDetector抽象类中已经定义并实现了的方法forward根据传入参数是否有return_loss来决定的 (PS: 这个调用又是基于torch中的nn.Module.call所导致的)

关于TwoStageDetector类中test函数的流程:

PS: 无论 aug_test 还是 simple_test, 两者的流程其实是相似的

  1. 会通过 extract_feat 函数获取到feature map
  2. 使用 rpn 模块相关函数进行 proposal list 的提取
  3. 通过 bbox 相关模块去进行 bbox、label 的获取
  4. 最后返回 bbox 的最终结果

关于TwoStageDetector类中train函数的流程:

  1. 会通过 extract_feat 函数获取到feature map
  2. 包含rpn部分的时候, 使用rpn模块进行处理, 得到proposal list结果以及rpn部分的loss
  3. 进行GT与proposal的匹配区分出来正负样本, 然后进行采样
  4. bbox head部分进行处理并获得classification以及bbox regression部分的loss
  5. 如果有mask部分进行mask部分的处理以及loss
  6. 将最终的loss进行返还.

Simple Summary for Representation Learning by Learning to Count

Posted on 2019-11-19

简单总结《Representation Learning by Learning to Count》

PS: 在论文总结中说的是本文学习在downsampled images与tiled images中数features数目并且使其在两者中满足等价关系.
PS: 另一个论文总结中说实质上本文受益的是: 在高层次上, 通常语义信息丰富、纹理信息复杂的norm都大

Introduction:

  1. 本文目的: 提出一个基于视觉原语的使用artificial supervision signal的方法(这里的artificial指的应该是人去定义的而不是人工标注的)去进行representation learning.
  2. supervised signal是通过equicariance relation获取的, 不需要人工的标注. 文章将transformations of image和transformations of representations相联系起来. 但是文章强调自己关注的找到满足这种关系的representation而不是针对特定representation的transformations. 本文在context of counting中使用两种transformations: scaling && tiling. 强调scaling利用的是visual primitives的珊瑚粮应该是与scale无关的, tiling则允许我们去同等看待整个图片中没个tile的visual primitives的总数. 提到了这两个transformations会被结合到一个约束中, 并用来使用一个contrastive loss去进行neural network的训练. 本文提出的方法在迁移学习中有很好的效果.
  3. 本文认为先前的自监督的工作表明了, 自监督学习的本质与最终要解决的问题最相关的自监督任务(pretext task)更可能构建出来好的representations. -> 本文的pretext task: counting visual primitives (这个自监督信号需要有辨别力的feature, 对分类是有帮助的, 同时其可以通过detection任务进行formulated).
  4. 为了获取对counting有利的supervision信号, 利用了如下的性质: 如果我们将image分割成了不重叠的regions的时候, 每个regions 中 visual primitives的数量加起来就是整个图片中visual primitives的数量
  5. 同时本文做出的假设是: 模型需要能够将image分离处high-level factors of variation,从而将orginal image与其regions之间的复杂关系转变为简单的算数关系(本文就是counting).
  6. 强调了虽然本文使用的是特定的两种变换的组合, 但是可以使用其他的变换的组合作为vision signals. 同样可以用于更大范围的任务之中.
  7. 本文贡献: (1). 引入了一个新的不使用手工标记数据的特征学习方法 (2). 提出利用counting作为自监督任务, 同时证明了其与counting visual primitives之间的联系(???) (3). SOTA效果

Method:

  1. Transforming Images to Transform Features:
    • (1). 提及到characterize a feature of interest的一种方式就是描述其是如何作为function随着input data的变化而变化的. eg: visual primitives的计数不应该被scale, 2D translation, 2D rotation变化所影响. 整体的magnitude of feautres应该大于region的.
    • (2). 本文, 作者会使用已知的input与output之间的关系作为superisory signal 去进行representation的学习.
    • (3). 首先定义了图片为 $x ∈ R^{mn3}$ 定义了一组image transformations $G={G_1, …, G_J}$ 每个变换 $G_j : R^{mn3}->R^{pq3}$ 将image作为输入, 并且将其映射到 p q大小的image上. 之后定义一个feature $φ:R^{pq3}->R^k$ 将transformed image映射到一个 k 维度的vector上. 最后, 定义一个feature transformation $g:R^k···*R^k->R^k$ 将J个features作为输入, 并将其映射到另外的一个feature中. 因此给定了image transformation family G 和 g, 通过后面的公式所定义的artificial supervisory signal学习feature φ: $g(φ(G_1◦x), …, φ(G_j◦x)) = 0$
    • (4). 在本作中, transformation family由downsampling operator D(下采样系数为2), tiling operation operator $T_j, j = 1, …, 4 $组成. 因此 $G={D, T_1, … ,T_4}$. 同样定义了 $g(d, t_1, …, t_4) = d - Σ_{j=1}^4t_j$ 这个公式可以被写成 $φ(D◦x)=Σ_{j=1}^4φ(T_j◦x)$
    • (5). 说这个关联也和equivariance(同变性, 指的是不要求CNN所强调的invariance不变性, 而是要考虑到坐标之类的信息的)有所关联. Equivariance通常被定义为一个给定feature的性质. 本文中通过固定transformations并寻找一个满足这些变换的representation来将上面的定义的逻辑反过来. 同时, equivariance被约束在应用在inputs和features的transforamtions上. (???有点迷)
  2. Learning to Count:
    • PS: 使用CNN去获取representations.
    • (1). 原则上可以去用一个large database中的图片进行训练, 使用基于上面counting公式的 l2 loss作为损失函数. 即: $l(x)=|φ(D◦x)-Σ_{j=1}^4φ(T_j◦x)|^2$ 但是认为loss总是会有 φ(z) = 0 的这个平凡解的. 为了避免这个情况的出现, 选择使用contrastive loss, 连接是知乎一个描述 本文中的 loss形式为: $l_{con}(x,y)=|φ(D◦x)-Σ_{j=1}^4φ(T_j◦x)|^2 + max{0, M-|φ(D◦y)-Σ_{j=1}^4φ(T_j◦x)|^2} , x\not=y$ 目的是希望能够强迫两个随机选择的不同的image的counting feature尽可能的不同. M取经验值为10.
    • (2). 结合结构图食用可以明白: 整个流程是: 1). 选取两张不同的图 2). 将一张图只进行downsample, 将另外一张图做downsample并且进行tiling 3). 使用共享参数的AlexNet进行2).中得到的图的处理 4). 每个后续接上几组全连接层进行处理, 得到对应的最后的vector 5). 进行Loss的计算, Loss分为两个部分 1. 进行了downsample以及tiling的图片中: 计算正常的counting loss的计算 2. 只进行downsample图片的最后结果和另外image的tiling计算结果进行max(0, M - counting loss)的计算(希望两个不同的image 的 intra-image 差异尽可能的大)
    • (3). Least effort bias: a bias of the system是: 可以通过学习尽可能少的对visual primitives进行count来达到基本loss(指的是$l(x)=|φ(D◦x)-Σ_{j=1}^4φ(T_j◦x)|$)的目标 因此很多的feature mapping会坍塌到0, 在imageNet、COCO等数据集上已经发现了1000个entry中只有几十个非零, 特征因此是稀疏的. 此外本文的transfer learning实验表明了hidden layers在一些benchmarks上表现很好. 但是本文的contrastive loss限制了least effort bias. 因为counting feature当可数的visual primitives很少的时候并不能很好的和其他图像进行区分. 因此contrastive项也可以看做是将不同图像(intra-)区分开来的作用.

Simple Summary for Deep Clustering for Unsupervised Learning of Visual Features

Posted on 2019-11-18

简单总结《Deep Clustering for Unsupervised Learning of Visual Features》

PS: Unsupervised 方法(Self-Supervised), 在综述贴中, 描述这篇文章描述的主要是 利用对feature representation的clustering去进行parameter的更新

PS: 确实本文的方法是无监督的方法, 通过聚类的方式产生了 伪标签去进行训练来得到好的representations, 但是最后的结果是怎么产生的??? 聚类类别明显是很多的啊, 最后是怎么对应到各个类别的呢? 而且怎么用来去做分割已经detection也不是很明白… 是后面要进行有监督的学习进行fine-tune吗? 那使用VGG什么的带来的效果看上去并没有那么好.

Introduction:

PS: 总体流程上个人认为还是整个pipeline, 在feature提取层上取出来对应的提取的特征, 用来进行无监督的分类, 然后给上一个伪标签, 后续进行到分类网络的最后进行使用这个伪标签的训练去反向传播(图并没有画最后面的一部分)

  1. 想要将clustering以一种end2end的方式融合到深度学习网络中, 并且是其能够基于大规模的数据集进行训练. 本文提出的是DeepCluster, 一个联合学习神经网络参数和对得到的features结果进行聚类标签的聚类算法. DeepCluster迭代式的使用标准聚类算法(K-means)对features进行聚类, 并使用subsequent assignments作为监督去进行network的权重的更新. 使用DeepCluster去无监督的对CNN在大数据集上进行训练. 最后可以达到SOTA效果.
  2. 从Fig 1中可以看出的本文的主要做法: 将没有标签的数据进行clustering做分类, 之后形成伪标签, 然后用伪标签去进行训练.
  3. 提到了传统的聚类算法通常是针对具有固定feature的线性模型进行的设计, 这些工作也很少能在学习features的同时很好的工作.
  4. 本文, 作者提出了新颖的聚类算法, 针对大规模的convnets的end2end的训练. 表明了模型能够使用聚类网络获取到有用的通用features. 本文的做法就是: 交替的进行(1). 对图像描述子进行分类 和(2).通过预测cluster assignments进行convnet的权重的更新. 文章说是为了简单才选择使用的Kmeans, 其他聚类算法也是可行的. 认为与self-supervised方法不同的是, clustering只需要很少的domain knowledge, 并且不需要inputs的特定signal.
  5. 本文同样通过修改实验规则证明了框架的robustness(修改了包括convnet网络结构以及数据集), 也强调了不仅仅能捕捉class-level information, 同时也能捕捉instance-level information.
  6. 主要贡献点: (1). 一个新颖的使用如k-means的基础聚类算法的end2end训练的unsupervised方法, 同时仅仅需要很少的额外步骤. (2). 在很多的unsupervised transfer任务上SOTA的性能. (3). 一个关于unsupervised feature learning的验证标准的讨论.

Method:

  1. 首先对一些流程、目标与定义进行了公式化的描述
  2. Unsupervised learning by clustering:
    • 首先强调了, 当对提取特征的network参数进行随机初始化的时候, 并不会产生好的features, 但是random features的性能要比瞎猜要好多了. 认为这个好的效果是由于conv本身的结构所带来的对input signal的强先验导致的. 本文的idea就是去利用weak signal去促进convnet的辨别能力.
    • 本文对convnet的output进行cluster, 然后使用后续的cluster分配的伪标签进行最后损失函数的更新, 这个过程会迭代的进行去进行features的学习然后进行分类
    • 这里指的convnet应该是没有接上后面进一步辨别网络的部分, 只是用来学习特征的, 文章使用k-means作为基本的聚类算法(但是作者观点是按照以往的工作, 这里使用什么聚类算法都是无所谓的)
    • k-means使用的分类标准是几何距离
    • 引入了一个 d * k (d是feature的维度, k是分类的类别)的centroid matrix C, 通过 $min||(f_{θ}(x_n))-Cy_n||^2_2$ 其中$y_n$的每一个entry都是0/1,且只能有一个1. 对这个问题进行优化来得到最佳的伪标签y_n以及矩阵C(虽然后续C没有使用)
    • 总体上就是, $min||(f_{θ}(x_n))-Cy_n||^2_2$ 去得到伪标签, 然后去进行分类任务的损失函数的更新进行网络参数的优化.
    • 但是提到了这么操作很可能会产生平凡解, 需要一定的操作去避免优秀解的退化
  3. Avoiding trivial solutions
    • 强调了平凡解(应该指的是所有的点都被分到了一个类别, 或者大部分的点在少数的几个cluster中的时候)的而存在, 不仅仅是对非监督训练neural network来说的, 对于任意联合学习discriminative classifier和labels的方法都存在平凡解的问题. 提到了discriminative clustering即使在线性模型中使用的时候也会出现这个问题. 提到了解决办法通常是: 约束或者惩罚每个cluster中的点的数量, 但是提到了因为涉及到了整个dataset中的计算处理, 不适合用于大规模数据集上的处理办法, 所以提出了一系列其他的想法:
    • (1). Empty clusters: 提到了出现所有点在一个类别的可能性之一在于没有限制empty cluster的产生. 提到了一个最常用的方法就是自动的去重新分配empty cluster -> 当一个cluster是空的时候随机选择一个非空cluster并且将这个非空cluster的中心添加一个小的随机扰动之后作为空集的新的中心, 然后重新的去将非空集合按照这两个中心点进行一次聚类
    • (2). Trivial parametrization: 提到了当分类时候将大部分的image分配到了少部分的cluster中时, 学习到的parameter θ会对这几个类别非常有辨别能力. 尤其是当极端情况下只有一个类别的时候, 这个时候分类类别将会百分百正确. 这种情况通常在数据类别不平衡的时候出现. 避免这个问题出现的一般策略就是从uniform distribution中进行image的sample, 或者使用伪标签.
