Simple Summary for FoveaBox: Beyond Anchor-based Object Detector

Posted on 2019-12-18

简单总结《FoveaBox: Beyond Anchor-based Object Detector》

Introduction:

  1. 本文–FoveaBox 直接学习物体的存在概率以及 bbox 坐标, 而不依赖于anchor(PS: 说人话就是 anchor-free)
  2. 本文实现使用了:
    • (1). 预测 category-sensitive 语 maps 去判断物体存在的概率分布.
    • (2). 使用 category-agnostic bounding box 去对每个可能包含物体的位置预测 bbox.
  3. 提到了 FoveaBox 的泛化性比较好, 能够对 bbox shape 分布的改变有很强的鲁棒性和泛化性.
  4. 本文也提及到了 anchor-based 的缺点 -> (1). hyperparameter的增加 (2). 不同 dataset 下 anchor 的设置并不通用 (3). 因为密集采样去将 anchor 尽可能的遍布整个 image, 所以需要技巧去处理 imbalance 问题
  5. 提到了现在的一些文章通过改进 anchor 上的成功的做法:
    • PS: 下面三篇文章都没有看过… 描述应该不是很准确
    • PS: 下面改进这方面的思路都是使得anchor更加灵活
    • (1). MetaAnhocr: anchor functions 动态的根据任意的 prior boxes 生成.
    • (2). Guided-Anchoring method: 联合的预测 1). 可能分布了物体的中心的概率, 2). 不同位置处的 scales/aspect ratios
    • (3). Anchor box optimization for object detection: 动态的学习 anchor 的形状
    • 但是文章认为, 这些 works 还是依赖于枚举所有可能的 scales 以及 aspect ratios.
  6. 本文的动机:
    • PS: 又一个说是根据人眼的想法启发的… 人眼的 fovea(好像是虹膜)
    • (1). 也是强调 vision field 的中心区域是 最敏锐的 -> FoveaBox 联合预测 1). 位置是物体中心的概率 2). 每个有效位置的 bbox 信息
    • PS: 本文设计的是 detection head (基于 feature pyramid)
  7. 为了验证本文提出的 anchor-free 方法的有效性, 将数据进行拉伸处理, 同样可以取得一个很好的效果, 大幅度超过了 anchor-based 的方法.

Method:

  1. 总体上的描述: backbone之后会有两个 subnet(或者可以称之为 两个分支), 一个用来预测每个pixel的classification 一个用来进行 bbox prediction. 基本结构和很多文章一样同样使用的是RetinaNet
  2. Feature Pyramid Network Backbone:
    • (1). 使用 FPN 作为backbone, 也是先简单对 FPN 进行了一下描述
  3. Scale Assignment:
    • (1). goal -> 预测 target obj 的边界.
    • (2). 本文同样是按照尺寸去进行 pyramid level 的分配的, 但是与之前不同的是, 之前的做法是: 这个范围是不兼容的, 也就是一个 proposal 只能在一个 pyramid level 进行训练. 本文的做法是: 层次之间是有重叠的, 一个object可以在多个 pyramid 中进行训练 -> 这个就是 SPAD 提到的多个层次同时训练有时候是有好处的.
  4. Object Fovea(实际上就是 类别的 heatmap):
    • (1). 预测的实际上是四个坐标值, 看上去好像和 FSAF 的距离四条边的距离有一点不同
    • (2). 定义的 positive 区域实际上是中心的一部分区域, 其余向外扩充一圈的区域是 忽略掉的, 剩下的部分作为 negative. 因为 positive 相比于 negative 来说总是很少的, 所以使用 focal loss 去进行训练. (这些都是和 FSAF 一样的)
  5. box prediction:
    • (1). 就是进行相当于之前的 bbox regression 阶段
  6. Ablation Study:
    • (1). Various anchor densities and FoveaBox: 使用 RetinaNet 作为 backbone 去不断的调整 anchor scale 以及 aspect ratio 的数目组合, 和 FoveaBox 进行对比. -> anchor 的不断变密集一定程度上是有利于性能的, 但是是会有一个上限的, 且并不能和本文的 anchor-free 方法的性能达到同一个档次.
      • PS: 给出的一个解释是: 1). (个人猜测)可能会加剧 fg/bg 的 imbalance 问题 2). 会造成 fg/bg 的难以回归 3).造成位置定位的模糊性->文章说不单单要对每个位置生成的 anchor 进行判断是否该位置的anchor是否是好的, 还要对每个文职的不同anchor进行区分, 然而单个 feature pixel 是使用相同的 feature 去进行信息的利用的, 这样的强行区分就会导致ambiguous. (个人认为也应该是因为很多anchor可能都是好的选择, 在这些anchor中间选择可能会带来模糊性而影响性能.)
    • (2). Analysis of Scale Assignment: 调整文章中设置的用来分配层次的参数的值, 将单个的 instance 可能分配到不止一个层次上的时候 进行了一定的实验, 分配到多个层次上有利于, 但是分配到太多层次上会带来不好的效果. (PS: 结合 SPAD 可以认为确实原来的按照尺度进行分配的策略是有一定问题的… 但是怎么解决…)
    • (3). 剩下的部分就自己看吧

问题:

  1. 总结一下就是大部分还是anchor-free(anchor-like)比较传统的设计思路, 后续很多的改进也是基于其进行的(比如FSAF), 主要有用的点还是应该算是那个单一instance多层次 assign 的设计.

Simple Summary for Learning Rich Features at High-Speed for Single-Shot Object Detection

Posted on 2019-12-15

简单总结《Learning Rich Features at High-Speed for Single-Shot Object Detection》

Introduction:

  1. 本文首先说, 最近的研究表明了, 从头训练 detection 网络是有利于减少 classification 和 localization 之间的 gap的, 尤其是在 high overlap thresholds 上. 但是从头训练 detection models 所需要的时间远远超过了使用基于预训练模型进行训练finetune的情况. -> 本文作者提出了一个新的 single-stage detection framework, 其能够结合 fine-tuning pre-trained models 和 training from scratch 的好处. 本文的模型结构包含了一个使用预训练 backbone 的基本网络和一个并行的 light-weight auxiliary network (这个网络部分是 train from scratch 的).
  2. 进一步的 文章说明, 仅仅使用的 top-down pyramid representation 只能够关注于 将 high-level 语义从 top layer 传递到 bottom layer 上. 本文引入了一个 bi-direction network, 能够有效的将 low/mid level 和 high level的 语义 信息循环使用.
  3. 本文在 相比于 SSD这种上有着很大的提升 将近10个百分点. (PS: 因为基于的是 SSD 这种 small input 的… 300*300 input size) 很不错了, 主要是足够快.
  4. 对小物体的检测效果很好, 算是部分的解决了小物体检测上的痛处.
  5. 确实在 Introduction 中说明了, 目前很多性能好的 one-stage 方法在小物体的检测上的表现还是很不好的. -> 强调小物体的检测需要 low-/mid-level 的信息去准确定位以及 high-level 语义信息去区分目标物体
  6. 认为当前的 object detectors 都是使用 top-down pyramid feature representation, high-level information from top layer 会与 weaker high-resolution 的 low layer feautre 进行 fuse.
  7. 强调了, 虽然传统使用的 top-down feature pyramid representation 通常都会产生一个不粗的结果, 但是其仅仅拒绝的是 high-level semantics to former layers. 本文则 两边的(高层向底层, 底层向高层) 的进行融合, 并表明了这么做对于多尺度 object detection 是十分有作用的.
  8. 本文主要贡献:
    • (1). 首先是本文提出了一个轻量级的 从头训练的网络(LSN), 其使用 downsampled image 作为 input, 并且将其传递给一些 conv 层以构建 low-/mid-level features.
    • (2). 上面的轻量级网络 LSN 将会嵌入到整个detection中, 并且作者引入了 bi-directional network 去 circulate both low-/mid-level and high-level 语义信息.
  9. 文章简单的描述了一下标准的 SSD, 然后说明了一下其缺陷:
    • (1). 每一个层次都单独进行 detection. 对 large scale variations 来说难以处理, 尤其是针对小物体的时候. -> 认为是前面的层次相比于后面的层次来说缺少语义信息
    • (2). 还是强调了top-down feature pyramid情况下 前面的层次不能利用到后面层次的语义信息, 语义信息不够. 而加深网络/感受野等方法在提高精度的同时引入了大量的性能消耗

Method:

PS: 首先放一张结构图…

  1. Overall Architecture:
    • (1). 整体结构其实相对来说思想很简单, 首先不谈原始的 SSD Backbone. 先是引入了 一个轻量级的从头训练的网络(个人认为 1. 参数量少, 好从头训练 2. 轻量级, 速度快). 这个分支提取出来的小的 feature map主要是用来和原本的 pipeline 的 feature map 进行融合(bottom-up融合) 之后会通过 top-down 结构再次进行信息融合(这一个 top-down 一个 bottom-up 就是所谓的 bi-derectional network 了)
    • (2). 对比了一下 bi-directional network 和 FPN:
      • 1). 首先强调自己的 bottom-up feature pyramid 的生成是 cascaded 式的, 而不是像 SSD/FPN 一样只是简单的 forward 获得的
      • 2). 然后表明了自己 top-down pyramid 不是像 FPN 一样 layer-by-layer 进行fusion的, 而是并行连接(指的是每个 top level feature map 都会直接提供自己的信息去给低层进行 feature fusion)
  2. Light-Weight Scratch Network:
    • (1). 首先提到了 detection frameworks 中常用的 feature extraction strategy -> 从 backbone 中提取 features. 又是强调了一下这个做法的不妥之处. -> 本文提出的 LSN 网络结构就是为了解决这个问题的
    • (2). LSN 网络的几个特点: 1). input image 是通过 pooling 操作 downsample 下来成 SSD 第一个 prediction layer size的. 2). downsampled image 会通过一系列轻量级的操作进行处理 3). LSN 是随机初始化去进行from scratch训练的. 然后可以得到对应的小的 feature pyramid 4). feature pyramid的每一层的尺寸都是和 SSD feautre pyramid 相同的(为啥轻量级就是因为这个是直接从很小的原图进行特征提取什么的…)
    • (3). 网络的具体结构什么的看图就大概可以明白了.
  3. Bi-directional Network: 同样没什么特别的地方… 除非要仔细的研究一下结构是怎么设计的.

问题:

  1. 所以总的来说文章想要解决的就是: 同时使用 top-down && bottom-up 的 feature fusion, 同时保证计算量不怎么变化. 引入的 轻量级结构实际上是用来进行 feature fusion 轻量级化的…… 感觉什么从头训练只是一个噱头…

Simple Summary for Soft Anchor-Point Object Detection (SAPD)

Posted on 2019-12-13

简单总结 Soft Anchor-Point Object Detection (SAPD)

Introduction:

PS: 本文是 FSAF (Feature Selective Anchor Free) 那篇文章的大佬的作品… 本文效果超级好 再一次拉高了标准(难受) 膜

  1. 提及了最近的 anchor-free 方法在精读和速度上都大幅超过了 anchor-based 方法. 本文的目的是去寻找 anchor-free detector 的 speed 和 accuracy 的 trade-off/平衡点
  2. 关注于两个问题:
    • (1). 如何能够使得 anchor-free detection head 更好
    • (2). 如何能够更好的利用 feature pyramid (继续深挖feature selective)
  3. 解决这两个问题的角度: (1). attention bias (2). feature selection
  4. 本文提出的是训练策略(果然…) -> 包含了两个 soften optimization techniques
    • (1). soft-weighted anchor points
    • (2). soft-selected pyramid levels
  5. 精度-速度的trade-off达到了一个更高的高度 -> 同等精度下, 可以达到 5 倍以上的速度
  6. 再次强调了 anchor-free detector 的优势:
    • (1). 不需要手动的调整 anchor configuration 的 hyperparameters
    • (2). 通常来说有着更加简单的 detection head 的形式
    • (3). 在精读和速度上都超过了 anchor-based 方法
  7. 同时分析了两类的 anchor-free detector (1). Anchor-point detector (2). Key-point detector
    • (1). 提到了对于 Key-point detector: 来说其通常能够在 small input image size 的时候达到一个相对来说较高的 AP. 但是其通常依赖于单一的 high-resolution feature map, 并且重复 bottom-up top-down inference -> 导致了高 FLOPs 以及 计算量、训练以及测试时间. 并且与主流的预训练 backbone 相对是低兼容性的.
    • (2). Anchor-point detector: 通常有着简单的网络结构, 快速的训练速度和推理速度, 鞠永送feature pyramid更好的受益的潜力(应该是因为key-point based detector通常都是基于Hourglass-101的单scale), 灵活的 feature level selection. 但是在特定的image scale testing情况下是没有key-point-based methods准确的, 尤其是定位准确性上.
  8. 强调一个问题就是 -> 是否一个简单的 anchor-point detector 可以达到和 key-point detector 相似的精读. -> 本文提出 SAPD
    • (1). 将低效的训练策略看做最主要的阻碍 anchor-point detector 性能的因素
    • (2). 两个没有被关注过的问题: 1). attention bias 2). feature selection
    • (3). attention bias -> 有着很好的 views 的物体将会倾向于获得到 detector 的更多的关注, 使得其他的物体容易被忽略(因为训练过程中的anchor-points misalignment).
    • (4). feature selection -> 自适应的去调整 instances 在指定的 pyramid level上训练 -> 更好的挖掘利用 feature map
  9. 本文的两个做法的大概描述:
    • (1). 基于到object center的距离去reweight positive anchor point的对 loss 产生贡献的权重
    • (2). soft feature selection weights去进行层次的分配 (用 meta-selection network 去进行分配)

Method:

  1. Preliminary on Anchor-Point Detectors: 首先对 anchor-point-based baseline(作者上一个作品FSAF) 进行了回顾: 这部分回去看 FSAF 就好
  2. Soft-Weighted Anchor Points:
    • (1). Attention bias: 强调了当面临着类似于 occlusion, cluttered background 等具有挑战的情况下, vanilla anchor-point detector 会存在 attention bias 问题. -> 有着很清晰的表现或者引人注目的表现的物体通常来说都是会产生一个很高的分数, 抑制了周围的物体, 甚至可能覆盖掉周围物体的表达. -> 导致了过分关注 foreground instances 抑制了 background instances.
    • (2). 问题的原因: 认为是因为 在靠近物体instance边界的位置上因为feature misalignment所导致的位置不确定使得边界周围的值 不必要的过高 所导致的. -> 所以在物体边界的其他的 anchor-point 会因为没有足够的与其对应的 instance 所以导致效果不好.
    • (3). 认为是因为 instance 感受野包含了太多的 background 信息, 所以导致了 representation power 的缺失
    • (4). 因此不应该将物体 center 附近和 boundary 附近同等程度的对待 -> 走 CenterNet 的思路了… 看上去好像没什么区别… (PS: 细说的话还是有的, 其自由度相比来说更强, 但是可能会牺牲一些东西…)
    • (5). 解决方法: soft-weighting scheme
      • 1). 主要思想: 对每个 anchor point 都给一个权重, 对每个 positive anchor point, 这个权重将会依赖于 其 image space location 与 instance 中心 的距离.
      • 2). 这个距离越长, 权重越低
      • 3). 对于 negative anchor point 来说, 所有权重都还是 1 不会变化
      • 4). 对于 soft-weighting scheme 会有几种选择. 本文选择… 看论文吧
  3. Soft-Selected Pyramid Levels:
    • (1). Feature selection: 提到了与 anchor-based detector 不同, anchor-free module 并不会通过约束 anchor matching 去进行 instance feature level 的选择 -> 可以将每个 instance assign 到任意的 feature level
    • (2). 首先其对 feature maps 进行了观察, 发现了不同层次的 feature map 是相似的, 尤其是相邻层次上的 feature map. 如果一个区域在某一个 feature map 上被激活了, 那么在相邻层次上的该区域也应该是被激活的, 但是随着层次的变远, 相似程度会下降 -> 多个层次的 feature map 可以一起去对检测一个特定的 instance 产生贡献, 但是不同层次贡献肯定不同, 要加权.
    • (3). pyramid level selection 应该有的两个原则:
      • 1). 首先要根据 feautre response 的 pattern 进行选择. -> 认为instance-dependent loss 是一个很好的反应一个物体应该属于哪一个层次的东西.(FSAF思路就是这个样子)
      • 2). 我们应该允许来自于多个尺度的 feautres 在训练和测试中对每个 instance 产生作用. (PS: FoveaBox表明了将instance分配给多个 feautre level 是有利于提升性能的.)但是强调了如果将物体分配到了太多的层次也是会严重影响性能的. -> 作者认为是 hard feature level selection 所导致的. -> 作者提出的就是能够根据 instance 去进行每个 pyramid levels 的 reweight.
    • (4). 解决办法: 选择去训练一个 meta-selection 网络去预测 soft feature selection.
      • 1). 首先模块的输入是从各个 pyramid levels 中提取出来的 instance-dependent feature responses (文章中是通过 RoIAlign layer 进行提取的) (PS: 我之前也想过这么干, 但是对于anchor-based one-stage detector来说那个东西真的叫做detector吗…)
      • 2). 然后提取出来的 feature 经过 meta-selection network 然后产生一个 vector. 使用这个权重作为 soft feature selection.
      • 3). meta-selection模块 -> 3 个 3 * 3 conv层(无padding, 跟ReLU). 然后跟上一个 FC 层做 softmax(输出与通道数相同). 使用 cross entropy loss 去进行优化, label是根据 FSAF 最小的 loss 分支决定的 one-hot vector 进行优化的.
      • 4). 因此每个 instance B 都会有一个 权重(meta-selection的输出) 再结合上上一个的 soft-weight anchor-point, 就可以得到对应最终的该instance的权重.
      • 5). 网络的总Loss会包含两个部分, meta-net 部分的 loss 的 weight 是会调整的.

问题:

  1. 这个soft feature map是不是要好好想一下…
    • (1). 相当于一个二次的 feature selective, 但是好坏信号是靠 anchor-free 分支所决定的, 这个是不是有一些更好的想法(PS: 为什么要靠 anchor-free 分支去进行决定而不能考 anchor-based 分支(虽然本文是完全anchor-free的))
    • (2). 层次的 soft 怎么都觉得奇怪…
  2. soft label 相比于 centerNet 来说只是多了一下 anchor-point 并不是中心, 多了自由度. (PS: 这其实就是相当于向 centerNet 进行了靠拢… 因为标注的anchor还是那个anchor范围)
  3. 为什么相比于 FSAF 的 anchor-free 分支能有这么大的提升, 本质上的原因在什么??? FSAF 纯粹靠 anchor-free 分支的时候性能可是不太好的啊
  4. 加上 anchor-based 分支是会影响性能吗? 为什么? -> 如果是的话 猜测就是近期提出的那个 sample 策略上存在一定冲突.

(Chapter15--Exercise) Simple Summary for Very Deep Convolutional Networks For Large-Scale Image Recognition(VGG)

Posted on 2019-12-12

(Chapter15–Exercise) 简单总结(VGG) Very Deep Convolutional Networks For Large-Scale Image Recognition

Introduction:

  1. 本文的主要贡献: 验证使用 3*3 conv 的网络结构加深达到的效果
  2. 强调了之前的针对 (Krizhevsky的) Conv 网络的改进是 (1). 在第一个 conv 层使用更小的 conv 以及 stride. (2). 使用多尺度的方法进行network的训练以及测试(这里指的是 Overfeat, 和一些其他的当年的作品使用的方法)
  3. 提到了本文想要解决的是ConvNet结构设计的另外一个重要的方面: 深度 -> 将架构中所有的其他参数都固定, 然后仅仅去增加网络的深度, 堆叠更多的 conv layers. 认为是使用了小的 3 * 3 的卷积使这种做法变得可行.
  4. 本文贡献: 提出了一个新的更加准确的SOTA的深度convnet, 具有很好的广泛应用性, 且其提取到的特征使用简单的pipeline也可以得到一个很好的效果.

Method:

  1. 架构的一些设计:
    • (1). 224*224 RGB input image
    • (2). preprocessing -> 简单的减去平均图而已
    • (3). 使用small RF conv(3 3甚至是1 1的)
    • (4). 卷积前后保持尺度不变
    • (5). 尺度大小变化通过2 * 2max-pooling来做.
    • (6). 后续会使用3个FC层(4096-4096-1000)
    • (7). 所有的hidden layers都会使用ReLU进行激活
  2. 强调和主流数据集的主流设定不同的是, 本文提出的结构没有在第一个conv层使用大的kernel或者打的stride.
  3. 主张使用多个 3 3 的小的conv去在代替大的如 5 5、7 * 7 的conv. 认为好处如下:
    • (1). 多个卷积核后面跟上了多个激活单元, 意味着非线性的增加, 做出的决定是更加有辨别能力的.
    • (2). 减少了模型的参数
  4. 提到了适量插入 1 1 conv的目的是为了增加非线性的同时并不改变 Receptive Field. (提到了 1 1 conv是在 Network in Network中广泛使用的)
  5. 关于训练的一些设计:
    • (1). 使用带动量的mini-batch gradient descent进行训练, 动量为 0.9
    • (2). batch-size 调整到256
    • (3). 使用 L2 正则去进行权重约束
    • (4). 使用dropout regularisation(对前两个FC层)
    • learning rate 开始设置为 10^-2, 然后当在validation上的准确率不在增长的时候就减少十倍, 训练370K iteration(74 epochs)之后停止 -> 文章说本文的网络深参数多, 但是能够更快的收敛的原因是因为1). 网络的深度以及小conv kernel的使用带来的隐性约束 2). 特定层的预先初始化 -> 作者先使用一个比较浅的网络去进行训练, 然后提取这个网络的浅层(前4层以及后3层conv) 中间的使用随机初始化进行(PS: 同时训练的时候并不会减少对这些层的学习率) (PS: 作者提到了, 可以不按照这个方法, 而且是使用 Glorot&Bengio 提出的随机初始化方法代替这个方法也可以达到好的效果)
  6. Training image size:
    • PS: 设 S 作为各向同性放缩的最短边长度
    • (1). 如果将crop size设置为固定的 224 * 224 的时候, S 是要不小于 224 的, 当 S = 224 的时候, crop将会crop整个image. S >> 224 的时候只会 crop 出来 image 的很小的一个部分, 这个部分将会包含一个小物体或者一个物体的部分.
    • (2). 作者考虑两种进行 training scale S的设定: 1). 固定 S -> single-scale training. 2). 设置 S 为多尺度训练, 将训练图片随机放缩为 [S_min, S_max]之间
  7. Testing: 提到了为了不再对每个crop出来的image patch(因为之前前面是FC层)进行重复的的计算然后ensemble, 选择先对整个放缩好的 image 先进行特征的提取(7 7 conv + 2个1 1 conv) 得到一个 class score map(通道数是和类别数相等的, 并且是可变的空间分辨率的) -> 为了得到一个固定大小的关于image的class scores, 会先进性spatially averaged. 然后再用FC层去进行处理.
    • 但是后续还是会选择 crop 去提高 performance

  2. 进行 cfg.train_cfg.rcnn.sampler 进行 bbox_sampler 的构建
  3. 对 batch 中的每个 image 使用 bbox_assigner 进行 assign 操作
  4. 继续对 assign 操作得到的结果使用 bbox_sampler 进行 sample 操作 得到这个阶段的最后的结果

PS: 该部分正常的整体思路描述

  1. 每个 proposal 先赋值为 -1 (-1 表示不会care这些anchor)
  2. 某个候选框和所有 GT bbox 的 IoU 阈值都小于 negative IoU 阈值, 赋值为 0
  3. 对某个候选框, 如果与其有最大阈值的 GT bbox 的 IoU 阈值大于 positive IoU 的时候, 会将这个 proposal 的值赋值为这个 GT bbox 的编号
  4. 因为前三步是对每个 proposal 进行与 GT bbox 的匹配以及进行编号安排, 所以仍然可能存在 GT bbox 没有被匹配到, 所以要对 所有 GT bbox 再进行一次判断, 与其 IoU 最大的 proposal 将会被赋值为这个 GT bbox 的编号.
  5. 匹配完毕之后进行采样, 会按照一定的比例进行采样(due to the positive-negative imbalance problem)

      • 2). neg_iou_thr: 小于这个值的被看做为负样本
      • 3). min_pos_iou: 看做正样本所需的最小的 IoU 大小阈值
      • 4). gt_max_assign_all: 对于一个 GT 有多个 bbox 都与其有最高的 IoU 的时候, 是否将这些 bbox 都 assign 给这个 GT bbox.
      • 5). ignore_iof_thr: 忽略 GT bbox 的 iof 阈值(???什么阈值???), 当取负值的时候不会忽略任何 GT bbox
      • 6). ignore_wrt_candidates: 是否去计算 bboxes 与 gt_bboxes_ignore 之间的 iof
      • __init__ 函数中就是简简单单的进行 这些参数在类内的赋值
    • (2). assign 函数: assign(self, bboxes, gt_bboxes, gt_bboxes_ignore=None, gt_labels=None):
      • 1). bboxes: 就是所有在上一阶段整合处理之后的 proposals, (n, 4)的 Tensor
      • 2). gt_bboxes: 就是 GT bboxes信息, (k, 4)的 Tensor
      • 3). gt_bboxes_ignore: 忽略掉的 gt bbox
      • 4). gt_labels: GT bboxes的 Label
      • 首先会通过 bbox_overlaps 函数计算 overlaps
      • 然后会针对是否忽略框进行判断, 如果忽略的话进行一些操作(PS: 主要是筛选掉一些overlap项, 因为暂时用不到, 先跳过 233)
      • 之后就会使用得到的 overlaps 去进行 assign_wrt_overlaps 操作, 然后将得到的结果进行返回, 这部分结束, 继续看 assign_wrt_overlaps 函数…
    • (3). assign_wrt_overlaps 函数: assign_wrt_overlaps(self, overlaps, gt_labels=None)
      • PS: 这个就是真的按照上面的流程说的处理的了
      • 1). 首先就是使用 torch.Tensor.new_full 创建了一个 size 为 (num_bboxes, ) 的用 -1 填充的 Tensor, 其 dtype 为 torch.long
      • 2). 对每个 proposal 获取到有最大 overlap 的 GT IoU 值以及对应的 index -> max_overlaps, argmax_overlaps 分别代表着 最大的 overlap 值以及 index
      • 3). 对每个 GT bbox 获取到有最大 overlap 的 GT IoU 值以及对应的 index -> gt_max_overlaps, gt_argmax_overlaps
      • 4). 利用上面的到的参数 max_overlaps 对应 index 处的值如果小于 neg Iou threshold 那么就会赋值为 0 (PS: neg IoU threshold 也可以设置为一个 tuple 代表阈值在这个范围内的视为 negative sample)
      • 5). 进行 positive sample assign, 对所有 proposal 最大 overlap 的 GT IoU 大于 pos IoU threshold 的赋值为 对应的 GT bbox 编号
      • 6). 然后会对所有的 GT bbox 去找最大的 IoU bbox, 这个 IoU 如果没有超过 pos IoU threshold 的时候就放弃, 超过了就根据设置如果对所有相同 IoU 的都进行 assign 就对所有的对应的进行当前 GT bbox 编号的 assign, 否则就针对一个进行 (PS: 按照 torch.max 的逻辑, 这个通常都是最后一个…)
      • 7). 当传入的参数 gt_labels 非空的时候, 在这里就会 添加一个 assigned_labels 对应于该 proposal 对应的 GT bbox 编号类别 label.
      • 8). 最后返回的值会封装在 AssignResult 中
      • PS: em… 7)…
  2. 接下来对使用的 RandomSampler 进行分析:
    • (1). 首先对 __init__ 函数进行分析: 其会调用父类 BaseSampler 中的 __init__ 函数: __init__(self,num,pos_fraction,neg_pos_ub=-1,add_gt_as_proposals=True,**kwargs)
      • 1). num: 提取正负样本数量
      • 2). pos_fraction: 正样本比例
      • 3). neg_pos_ub: 最大负样本比例, 大于该比例的负样本会被忽略, -1 表示不忽略
      • 4). add_gt_as_proposals: 是否将 GT 加入 proposal 作为正样本
      • 操作: 只是简单的将参数进行赋值而已
    • (2). 那么接下来就对 forward_train 中使用的 sample 函数进行分析就可以了: 仍然是在父类 BaseSample 中实现的
      • sample(self, assign_result, bboxes, gt_bboxes, gt_labels=None, **kwargs):
      • 1). assign_result: 上一步进行 proposal 与 GT BBox assign 的结果
      • 2). bboxes: rpn forward 之后的 proposal 结果
      • 3). gt_bboxes: GT bboxes 信息
      • 4). gt_labels: GT labels 信息
      • 流程:
      • 1). 首先还是会从 proposal 中提取前四个维度(也就是 bbox 的信息, 不包含 score)
      • 2). 创建了一个 gt_flags Tensor, 暂时将所有的 bbox 对应位置都设置为 0 (表示不是 GT)
      • 3). 如果选择将 GT bbox 作为 proposal (即 add_gt_as_proposals == True), 那么会将 GT BBox cat进入, 并且会在 assigner 中加入这些, 然后将这些的 gt_flags 设置为 1 (表示是 GT) 拼接进去
      • 4). 使用 _sample_pos sample 出按照设置中数量的 positive samples, 代码中提到可能会有重复的 index 被 sample 出来(代码认为可能是 pytorch 的 bug), 所以要进行去重复的操作, 然后剩下的 samples 数量都是暂时需要的 negative samples 的数量了
      • 5). 如果在 config 中设置的 neg_pos_ub 非负, 按照这个比例设置 negative samples 数量的上限
      • 6). 使用 _sample_neg sample negative samples, 并且进行去重
      • 7). 将 pos_ind, neg_ind, bbox, gt_bboxes, assign_result, gt_flag 使用 SamplingResult 进行封装, 然后返回
  3. 最后我们将每个图片得到的 SamplingResult 实例放入 sampling_results 中, 这个阶段就暂时结束了.

二、RPN引入部分分析

  1. 首先, RPN 在 mmdetection 中是被看做为 head 部分的, 同样在第一篇总结文章中就有所陈述, head 部分指的是针对特定任务的网络结构部分.

  3. 总体相关的流程包括了几大部分:

    • (2). 将输出与GT信息传入 RPN 的 loss 部分获得loss函数

三、RPN_Head 类分析

  1. 首先看一下 RPNHead.__init__ 函数: __init__(self, in_channels, **kwargs):
    • PS: 可以看出, 只要求一定输入 in_channels 参数, 剩下的参数除了一个 类别数量 之外在其父类 AnchorHead 类的 __init__ 函数中都有默认的初始值, 所以 RPNHead.__init__ 只是赋值了两个必须填的参数值, 调用了 super(RPNHead, self).__init__(2, in_channels, **kwargs) 来进行初始化. 所以这个部分我们看 AnchorHead.__init__ 函数进行分析:
    • __init__(self, num_classes, in_channels, feat_channels=256, anchor_scales=[8, 16, 32], anchor_ratios=[0.5, 1.0, 2.0], anchor_strides=[4, 8, 16, 32, 64], anchor_base_sizes=None, target_means=(.0, .0, .0, .0), target_stds=(1.0, 1.0, 1.0, 1.0), loss_cls=dict( type='CrossEntropyLoss', use_sigmoid=True, loss_weight=1.0), loss_bbox=dict( type='SmoothL1Loss', beta=1.0 / 9.0, loss_weight=1.0))
      • (1). num_classes: 分类数. Faster-RCNN只是简单的分为前景和背景, 所以 num_classes 赋值为 2. (PS: 如果是 one-stage 的方法, 这里的类别通常就是81了)
      • (2). in_channels: 输入进来时候的通道数. 之前在 FPN 类的总结里已经看过源码了, 每个层次都会被卷积转化为 256 channel, 所以 Faster-RCNN 的 config 文件中这个值取 256
      • (3). feat_channel: (默认值为256), 表示特征层的通道数(就是根据 FPN 处理之后的 stage output 后继续使用一个 conv 进行处理, 这个 conv 处理之后 output 的 channel 数量), config 中是 256
      • (4). anchor_scales: (默认值为[8, 16, 32]), 代表??? config 中最后设置的值是 [8]
      • (5). anchor_ratios: (默认值为[0.5, 1.0, 2.0]), 每一个位置处各个 anchor 的 aspect_ratio
      • (6). anchor_strides: (默认值为[4, 8, 16, 32, 64]), 不同尺度 feature map 上每一个位置处, 一个 pixel 对应的 stride 大小.
      • (7). anchor_base_sizes: (默认值是None), 代表一个 pixel 占据的长宽大小是多少 值等于 sqrt(w*h) (PS: 这个值当取 None 的时候, 将会与 anchor_strides 相同)
      • (8). target_means: (默认值是(.0, .0, .0, .0))
      • (9). target_stds: (默认值是(1.0, 1.0, 1.0, 1.0))
      • (10). loss_cls: (默认值看上面吧), 表示 RPN 部分的 classification loss
      • (11). loss_bbox: (默认值看上面), 表示 RPN 部分的 bbox regression loss
    • __init__ 中涉及到的一些其他的流程如下:
    • (1). 首先还是会对各个传入的参数进行类内参数的注册, 然后会根据 loss_cls 中的 use_sigmoid 的值进行 cls 输出通道的判断, 会根据传入的 loss_cls, loss_bbox 进行损失函数的分别构建. 然后根据 anchor_base_size(实际上是 anchor_stride) 进行 anchor_generators 的填充, 最后会填充对应每个 stage 的 5 组 AnchorGenerator 对象用来之后的 anchor 的生成.
    • (2). 每一个 stage 的 anchor 的数量为 self.num_anchors = len(self.anchor_ratios) * len(self.anchor_scales)
    • (3). 最后会构建一下 RPN 阶段的 网络结构的卷积层, 使用 _init_layers 函数, 会构建三个 conv 层(RPNHead类中重写的是3个, AnchorHead类中是2个), 一个 rpn_cls, 一个 rpn_reg, 一个 rpn_conv. 具体信息看代码吧.
  2. 之后在运行的时候肯定会从 forward 函数进行传入, 为了并行的可能性, AnchorHead 中用 multi_apply 进行了处理, 但是先不管这个, 直接看调用的 forward_single 函数即可, 这个函数在 RPNHead 中是重写了的, 主要流程如下:
    • PS: forward 中的 multi_apply 还是要看一下的… 其是为了处理多尺度的, 因为 fpn 输出的是多个 stage 的, 需要每个 stage 进行单独的 forward_single 函数的调用
    • (1). 先使用 rpn_conv 进行处理, 然后用 relu 激活
    • (2). 使用 rpn_cls 处理得到 rpn_cls_score, 使用 rpn_reg 处理, 得到 rpn_bbox_pred
    • (3). 返回这两个结果(rpn_cls_score 和 rpn_bbox_pred)
    • PS: 一个问题: 根据 AnchorHead 中的代码可以看出, 对于 cls 分支的输出只有 9 个 channel ??? 而不是论文中的 18 个???

  4. 首先看 RPNHead.loss, 发现其仅仅是调用了一下父类中的 loss 函数, 只是将 gt_label 参数设置为了 None, 其余都是传递一下参数, 所以继续看 AnchorHead.loss:
    • 首先还是对参数进行分析: loss(self, cls_scores, bbox_preds, gt_bboxes, gt_labels, img_metas, cfg, gt_bboxes_ignore=None):
      • (1). cls_scores: 就是 rpn 阶段 cls 分支得到的结果
      • (2). bbox_preds: rpn 阶段 bbox regression 分支得到的结果
      • (3). gt_bboxes: gtBBox 信息

        • anchor_list: 就是上面生成的各个 image 各个 scale 的 anchorList
        • valid_flag_list: 同样是上面 get_anchors 生成的
        • gt_bboxes_list: TwoStageDetector.forward_train 中就传入的 GTbbox 相关信息

        • target_mean: Mean value of regression targets
        • target_stds: Std value of regression targets.

      • 2). 这里还会进一步的处理之后调用 anchor_target_single 去产生 anchor target 然后最后返回 (labels_list, label_weights_list, bbox_targets_list, bbox_weights_list, num_total_pos, num_total_neg) 具体流程暂时略过… 有需要的时候再回来看
    • (3). 得到了上述的一堆结果参数之后, 统计了一下 num_total_samples = num_total_pos + num_total_neg if self.sampling else num_total_pos 之后将得到的 labels_list, label_weights_list, bbox_targets_list, bbox_weights_list, num_total_samples 以及 rpn.forward 得到的 cls_scores 和 bbox_preds 和 cfg 传入 loss_single 中进行处理.

  5. 这个时候就拿到了 rpn_loss, 向 losses 中记录 rpn_loss, 然后从 cfg 中读取 cfg.train_cfg.train_cfg.rpn_proposal config 信息

    • (1). 首先还是会使用 grid_anchors 生成一下默认的 anchor 信息, 因为 anchor 的 refine 是 predict 系数的, 所以还需要原来的进行计算.
    • (2). 然后开始对每个 image 进行 单独的 proposal 的提取, 会先提取出来 img_shape, scale_factor 并且会对 rpn 输出的 cls_score_list, bbox_pred_list 进行 detach 操作, 剥离出计算图中. 然后将这些参数交给 get_bboxes_single去进行处理. 所以接下来要对 get_bboxes_single 进行分析

    • (2). 之后首先会选取 score topk的信息 (根据 cfg 中对应部分的 nms_pre 决定的)
    • (3). 然后筛选小于 cfg 中设定的最小尺寸的对象
    • (4). 然后 proposal 是 score(可以说是 confidence score) 以及 bbox 信息的concate.
    • (5). 之后使用 nms 进行处理, 得到最终的 proposal
    • (6). 将各个尺度的 proposal 在 mlvl_proposals 中 append 进去, 然后使用 torch.cat 进行多尺度的信息的拼接
    • (7). 如果设置中设置了要进行跨尺度的 nms, 那么进行跨尺度的 nms
    • (8). 筛选出最后的 cfg.max_num 个 proposals
    • (9). 最后将筛选出的 proposals 返回, 这就是一个 image 对应的 proposals 了, 多个 image 将会最后再 放入另外的一个 list 中, 然后返回, 我们就得到了 proposals 列表了

  2. 在 TwoStageDetectors 代码中, 训练时候前向传播过程中使用了 FPN 的部分在 forward_train 函数中的 x = self.extract_feat(img) 处, 查看 extract_feat 函数可以发现, 其就是单纯的先把 input feed 给 backbone, 然后如果网络是有 neck 的话就继续将从 backbone 中获取到的 ouput feed 给 neck, 所以, 我们只需要对 neck 部分是怎么进行的进行查看就可以了
  3. 可以看到, neck 是在进行整个 detector 的 build_from_cfg 的时候根据 cfg 传入参数去构建的, 即通过 FasterRCNN类 (实际上是其父类 TwoStageDetector) 的 __init__ 过程中根据 cfg.model.neck 进行 build_from_cfg 去构建的Neck部分, 同之前的总结, FPN 也是通过一个 Registry对象 进行注册进入的, 即 mmdet.models.registry.py 中的 NECKS, 所以接下来对 mmdet.models.neck.fpn.py 中的 FPN 类进行分析即可

三、FPN类的分析

  1. 首先看一下 FPN 类的 __init__ 函数是怎么进行模块的构建的吧: __init__(self, in_channels, out_channels, num_outs, start_level=0, end_level=-1, add_extra_convs=False, extra_convs_on_inputs=True, relu_before_extra_convs=False, no_norm_on_lateral=False, conv_cfg=None, norm_cfg=None, activation=None):
    • (1). in_channels: Backbone输出作为FPN输入的通道数, 看config文件中的设置应该是一个list, 然后看了一下ResNet50的原结构(实际上ResNet50, 101, 152的conv2_x,conv3_x,conv4_x,conv5_x模块中使用堆叠的block是相同的, 只是堆叠的模块的数量不同, 每个conv阶段输出的channel数量因此是相同的), config中设置的是 in_channels 是: [256, 512, 1024, 2048] 就是针对ResNet的 conv2, conv3, conv4, conv5 阶段的输出的channel数目. 也就是FPN论文中所说的 {C2, C3, C4, C5} 层, stride 分别为 {4, 8, 16, 32}
    • (2). out_channels: FPN输出的通道数, 由config文件可以看出是一个单一的值, 代表所有的feature层通道数目都是这个值 – config中设置为 256
    • (3). num_outs: outstage 的数量. config 文件中标出的是 5.
    • PS: Faster-RCNN配置参数中只涉及到了上面的三种参数, 下面的参数好像都使用的默认值
    • (4). start_level: 默认值是0.
    • (5). end_level: 默认值是-1
    • (6). add_extra_convs: 默认值是False. 表示是否使用 C6 层进行detection
    • (7). extra_convs_on_inputs: 默认值是True. PS: 没有看到使用
    • (8). relu_before_extra_convs: 默认值是False PS: 没有看到使用
    • (9). no_norm_on_lateral: 默认值是False. 是否在 lateral_conv 中使用 norm. False 的时候 lateral_conv 构建的时候 norm 就是 None. True 的时候会采用和 norm_cfg 相同的值
    • (10). conv_cfg: 默认值是None. 进行 conv 构建的时候的参数, 在 conv_module.py 中可以发现, 这个值好像就是用来选择 conv 的类型的. 默认使用的是 torchnn.Conv2d.
    • (11). norm_cfg: 默认值是None. 进行 conv_module 构建中的 norm 层构建的 cfg. 在 norm.py 中可以知道, 这个参数是用来选择 norm 层的类型的. 默认下(None)不会添加 norm 层.
    • (12). activation: 默认值是None
    • 开始进行 __init__ 函数的流程分析:
    • (1). 首先进行了各种class内值的赋值, 新增了 self.num_ins = len(in_channels) 参数, 其代表了从Backbone中提取阶段的数量. 然后还进行了一个判断: 当 end_level 为 -1 的时候, self.backbone_end_level = self.num_ins, 此时 backbone 结束阶段就是从 backbone 中提取阶段的总数, 否则 backbone 中提取的阶段总数 self.backbone_end_level 等于 end_level 的大小
    • (2). 从 start_level 的值开始到 backbone_end_level 的值结束, 进行 lateral_convs 以及 fpn_convs 的构建. 两种 conv 层的区别是: lateral_conv 是 1 1 尺寸的卷积核, norm 由 no_norm_on_lateral 进行调控, 没有 padding, in_channels 和 out_channels 大小不同, 就是由 __init__ 的输入的参数所决定的([256, 512, 1024, 2048], 256). 而 fpn_convs 是 3 3 尺寸的卷积核, padding 为 1, in_channels 和 out_channels 的大小都是 256.
      • 由此可以看出 fpn_convs 是 lateral_convs 处理 backbone 的各个 ouput stage 之后进一步进行处理的 conv 层.
    • (3). 会根据 add_extra_convs 参数以及 extra_levels = num_outs - self.backbone_end_level + self.start_level 决定是否添加额外的 conv 层. (PS: 代码里写了, 比如: RetinaNet(用了C6层)). 因此虽然按照 Faster-RCNN 的配置来说 extra_levels = 5 - 4 + 0 = 1, 但是 add_extra_convs 的值是 False, 所以并不会额外的添加 conv 层… 哦豁… 然而后面的 forward 可以发现, 虽然没有添加 conv 层, 但是 num_outs 参数还是会决定输出多少个 stage 的 output
  2. init_weights 就是进行 self.modules() 得到的所有 Conv 层进行初始化
  3. forward 函数:
    • PS: forward 函数是使用了 @auto_fp16() 装饰器的… 是强制使用 fp16 进行训练的, 整个网络本质上是混合 fp16 以及 fp32 进行训练的. 好像 resnet 部分就没有强制使用 fp16 训练.
    • (1). 首先 forward 函数只有一个参数: inputs, 这个 inputs 就是从 backbone 中得到的一组 output. 其channel 数量分别对应着 in_channels 各个 entry 的值的大小.
    • (2). 首先使用各个 stage 的 output 用 lateral_conv 进行处理 得到一组输出 laterals
    • (3). 按照 fpn 的上下层融合的思想 利用后一层的 upsample(这里使用的是nearest) 和当前层次进行进行加法, 对 laterals 进行更新
    • (4). 然后使用 fpn_convs 进行得到的 laterals 的 conv 处理, 得到了 outputs
    • (5). 接下来会进行 extra stage 数据的产生了, 对于 Faster-RCNN 等网络来说, add_extra_convs 是False, 只是会根据 num_outs 参数通过 max_pool 补上后面的 stage 的 output, 但是对于其他的比如 RetinaNet 等网络来说, 有 extra_convs 的话会进一步根据情况进行后续的 通过 conv 得到 stage output.
    • (6). 最后会将各个 stage 的 outputs -> outs 用 tuple 封装然后返回.

PS: 到此为止 FPN 类的解析就完全结束了

(二)、加载数据集:

mmdet/datasets/builder.py build_dataset函数:

  1. 传入的参数为: cfg(dict类型), default_args(暂时不知道这个是什么用, 一直传递但是一直没有怎么用过, 可能是为了以后扩展方便???)
  2. 通常来说传入的cfg就是一个数据库配置的参数字典, 但是其还是有其他的选择的, 几种选择如下所示:
    • (1). 传入一个为 dict 类型的 data cfg, 且其中的anno_file只涉及到一个数据集的名字: 会正常的使用 build_from_cfg 函数(来自 utils/registry.py 文件) 去构建数据集
    • (2). 传入的是 tuple或者list 类型的时候: 会将其一个个的拆分, 每个使用 build_from_cfg 函数去进行构建, 然后将这些得到的数据集, 使用ConcatDataset类进行封装处理
    • (3). 传入的是 dict 类型的data cfg, 但是anno_file涉及到了多个数据集的名字, 会使用 _concat_dataset 函数进行处理来得到最终的数据集
    • (4). 传入的是 dict 类型的 data cfg, 但是 cfg 中 type 参数为 RepeatDataset, 会使用 RepeatDataset 进行 cfg times 参数的次数的重复的组合

mmdet/datasets/loader/build_loader.py build_dataloader函数:

  1. 函数定义: build_dataloader(dataset, imgs_per_gpu, workers_per_gpu, num_gpus=1, dist=True, **kwargs):
    • (1). dataset就是之前用build_dataset构建的Dataset相关信息, 但是只是存储了anno_file等文件的路径作为参数, 并没有真正的将数据加载进来.
    • (2). imgs_per_gpu: 每个GPU分配的image数量
    • (3). workers_per_gpu: 每个GPU分配的提取图片用的线程
    • (4). num_gpus: 使用gpu的数量:
    • (5). dist: distribution缩写, 表示是否使用分布式
    • (6). **kwargs: 所有的这部分参数后面会传递进入Dataloader中, shuffle指的是是否打乱.
  2. PS: 针对分布式的先不管, 目前就针对单机进行分析
  3. batch_size = imgs_per_gpu num_gpus num_workers = workers_per_gpu num_gpus
  4. 使用的GroupSampler是基于pytorch的Sampler基础上实现的, 用来控制数据后续的sample. PS:其中调用了一个Dataset的参数flag: 看注释说flag是进行图片分组的, 图片aspect ratio > 1的flag为1, 否则flag为0
  5. GroupSampler初始化的整个流程
    • (1). self.group_size = np.bincount(self.flag) 是对两组(根据aspect ratio划分)的计数
    • (2). self.num_samples是两部分count分别除以samples_per_gpu(就是imgs_per_gpu)的向上取整再乘以samples_per_gpu, 只是为了保证数量是可以被整除的
  6. GroupSampler中的重写Sampler的函数:
    • (1). len(self): 就是简单的返回self.num_samples
    • (2). iter(self): 会对两个group进行分别的操作, 因为上面说了会为了保证每个组中的元素是被samples_per_gpu所整除的, 所以在使用的时候对对应的group的index数组进行shuffle之后 会用前面的部分来填补到想要保证的length
  7. PS: GroupSampler相当于只是对图片进行了随机的打乱, 方便后续的sample的随机性
  8. dataloader使用的就是torch提供的dataloader, 只不过sampler以及collate_fn是框架内实现的.
    • (1). sampler: 这个sampler就是刚才使用的GroupSampler这个东西。
    • (2). collate_fn: 这里是为了扩展到并行计算的时候使用的自定义的collate, 暂时略过, em… 是进行数据分配的
    • PS: pin_memory这个东西是选是否使用锁页内存, 在高端机器, 高内存剩余的时候选择设置为True即可

mmdet/utils/registry.py文件 中的 build_from_cfg 函数:

  1. build_from_cfg(cfg, registry, default_args=None): cfg是config dict, registry
  2. 之后还是会向要构建的Module传入cfg作为参数, 所以这个就需要config的参数和不同部分组成模块的统一设置
  3. PS: 本质意义其实是, 在test开始的时候其实已经引入了很多的类, 包含了各种各样的模型类可以进行调用, 然后使用这个函数去根据conf, 将类进行实例化出来

mmdet/utils/registry.py文件 中的 Registry 类:

  1. 其包含了两个参数: name以及module_dict. 前者是指定了该Registry类别的, 后者是一个 dict, 里面包含了该Registry中的module信息. 这个 dict 内部是 module_name: class 的形式
  2. 实际上name就是其类别, module_dict 是type: type对应class
  3. PS: Registry类框架中抽象出来了一些预先设置好的Registry, 然后后续会用Registry中的方法register_module做修饰器, 去使得各个模块一开始就加载进来了, 而不是后续去进行模块的创建.
  4. 重点: 刚开始看代码的时候, 发现在使用build_from_cfg函数进行dataset构建的时候, build_from_cfg要求Registry对象中必须要包含 cfg 中 type 名称的对应 class. 但是代码中构建数据集的时候传递的参数是 DATASETS, 其只是简单的设置了一个dataset作为Registry对象的name, 而module_dist中是没有任何赋值的. 所以按理来说是要报错的. 但是框架的处理方法是: 使用@DATASETS.register_module作为修饰器放置在每个数据库类型的Dataset Class上, 同时在package的init.py中将这些Class全部引入一遍, 此时 DATASETS 就已经注册了所有的Datasets Class(在 import 那个模块的任意东西的时候就已经导入了) 所以实际上就已经提前的引入好了

mmdet/models/builder.py文件 中的 build_detector函数:

  1. build_detector(cfg, train_cfg=None, test_cfg=None):
    • (1). cfg: 这个参数传入的是整个cfg文件中的cfg.model参数, 也就是模型相关的所有基本参数
    • (2). train_cfg: 这个参数是在训练中需要使用到的的参数, 为cfg文件中的cfg.train_cfg
    • (3). test_cfg: 这个参数是在inference中使用的参数, 为cfg文件中的cfg.test_cfg
  2. 实际上build_detector只是将dict进行整合之后, 调用了文件中的build(cfg, DETECTORS, dict(train_cfg=train_cfg, test_cfg=test_cfg))
    • (1). cfg: 是model的参数, 即cfg.model
    • (2). DETECTORS是抽象出来的一个Registry实例, 同DATASETS, 其在package的init.py文件中也会想DETECTORS中进行各种detector的注册
    • (3). build函数中其实就是将参数以及registry实例传递给build_from_cfg函数去进行构建, 即build_from_cfg(cfg, registry, default_args) 其中cfg就是实际上的cfg.model, registry就是DETECTORS
    • (4). FasterRCNN在这里就是TwoStageDetector的抽象, 什么都没有变更, 所以只需要看TwoStageDetector类的实现就可以了

(五)、FasterRCNN模型代码部分

PS: 就是看TwoStageDetector类的实现

mmdet/models/detectors/two-stage.py 文件中的 TwoStageDetector 类

PS: mmdetection框架将整个network分成了几个部分: mmdetection的github doc中说大致上分成了四个部分: 1. backbone: 就是网络前面的提取特征的backbone, eg: ResNet 2. neck: 在backbone与head中间的部分, eg: FPN, HRFPN(HRNet提出的), BFP(Libra R-CNN的那个), ASPP 3. head: 针对特定任务的部分, eg: bbox prediction, mask prediction 4. roi extractor: 从feature map中提取feature 的部分, eg: RoI align.

  1. init(self, backbone, neck=None, shared_head=None, rpn_head=None, bbox_roi_extractor=None, bbox_head=None, mask_roi_extractor=None, mask_head=None, train_cfg=None, test_cfg=None, pretrained=None):
    • (1). backbone: 网络的backbone部分的config dict
    • (2). neck: 网络的neck部分的config dict
    • (3). shared_head: shared_head部分的config dict(PS: 在shared_head module里只看到了一个ResLayer…)
    • 后面同理, 直到mask_head都是网络对应部分你的config dict
    • train_cfg, test_cfg仍然是训练时候和测试时候的config dict, pretrained是是否有预训练模型
  2. 之后在TwoStageDetector.init()中就都是使用 mmdet/detectors/builder.py 中的各个build函数进行各个部分的根据build_from_cfg的构建的(PS: 每个部分仍然是都有自己单独的Registry, 且在其部分module的init文件中已经被注册进入). 最后会进行pre_train weight的加载(只在train过程中有可能进行了加载, 在inference中因为test.py中设置了cfg中的model.pretrain=None, 所以是不会进行加载的, 而是在model建立之后才进行checkpoint的加载)
  3. 关于pre_train weight的加载, 除了backbone以及shared_head部分的初始化是使用pre_train文件中的weight了, 其余部分的初始化都是根据各个module中自身定义的初始化函数去进行定义的, 大部分使用的都是pytorch自带的normal_init进行的参数的初始化
  4. 其他函数:
    • (1). init_weights(self, pretrained=None): 就是使用pre_train参数进行model初始化的函数
    • (2). extract_feat(self, img): 从backbone+neck中进行feature的提取, 其实就是将input经过backbone以及neck部分的forward而已
    • (3). forward_dummy(self, img): 计算网络的flops
    • (4). simple_test(self, img, img_meta, proposals=None, rescale=False): 只使用单一尺度的tesst
    • (5). aug_test(self, imgs, img_metas, rescale=False): 使用数据增强, multi-scale的test
    • PS: 测试过程会在BaseDetector根据img的第一维度长度进行 aug_test还是simple_test的选择.
    • (6). forward_train(self, img, img_meta, gt_bboxes, gt_labels, gt_bboxes_ignore=None, gt_masks=None, proposals=None): 训练过程中向前传递的过程, 会返回loss, 具体部分后面分析
    • PS: (4). (5). (6). 三个函数实际上是在 BaseDetector抽象类中已经定义并实现了的方法forward根据传入参数是否有return_loss来决定的 (PS: 这个调用又是基于torch中的nn.Module.call所导致的)

关于TwoStageDetector类中test函数的流程:

PS: 无论 aug_test 还是 simple_test, 两者的流程其实是相似的

  1. 会通过 extract_feat 函数获取到feature map
  2. 使用 rpn 模块相关函数进行 proposal list 的提取
  3. 通过 bbox 相关模块去进行 bbox、label 的获取
  4. 最后返回 bbox 的最终结果

关于TwoStageDetector类中train函数的流程:

  1. 会通过 extract_feat 函数获取到feature map
  2. 包含rpn部分的时候, 使用rpn模块进行处理, 得到proposal list结果以及rpn部分的loss
  3. 进行GT与proposal的匹配区分出来正负样本, 然后进行采样
  4. bbox head部分进行处理并获得classification以及bbox regression部分的loss
  5. 如果有mask部分进行mask部分的处理以及loss
  6. 将最终的loss进行返还.
