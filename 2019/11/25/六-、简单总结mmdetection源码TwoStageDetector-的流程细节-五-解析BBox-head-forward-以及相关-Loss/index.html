(六)、简单总结mmdetection源码TwoStageDetector的流程细节(五) – 解析BBox head forward 以及相关 Loss

一、前言

  1. 上一篇总结介绍了关于 GT assign 以及 Proposal Sample 的整体流程, 这个部分就是 TwoStageDetector.forward_train 的最后的部分了(不考虑 mask 相关部分), 但是到这里还是有疑问没有解决的, 具体疑问看 3.
  2. 本篇总结将会对 BBox head forward 以及相关 Loss 部分(mmdetection代码的注释是这么写的)进行解析.
  3. 疑问就是从上一个总结就有的, 关于多尺度信息在训练的时候怎么去区分, 因为上一篇文章也注意到了, 进行 GT BBox 匹配的时候也并没有区分什么尺度问题, 都是根据 IoU 在进行的计算, 然后进行筛选, 仅由的针对多阶段的过程还是 RPN 部分的 rpn_head.get_bboxes 其将多尺度的信息直接并成了一个维度, 只是有一个选项是选择是否进行跨尺度的 nms 操作. 希望在本文的 roi_extractor 部分会涉及到如何进行尺度分配的部分
    • 问题找到答案了: 在 mmdet.models.roi_extractors.single_level.py 中的 SingleRoIExtractor 类的 forward 过程中 会进行 map_roi_levels 函数的处理, 会根据 RoI 的大小去分配对应的 stage
  4. 另外的一个疑问: 从这部分的代码中已经可以看出了, 提取出来的 features 是来自于各种不同 scale 的啊, 后续的分支是公用的吗???

二、总体流程

  1. 首先将上一个阶段得到的 sampling_results 中的每一个 img 的 SamplingResult 对象提取出来其中的具体信息( torch.cat([self.pos_bboxes, self.neg_bboxes]) ), 然后进行 bbox2roi 操作, 就是添加了一下所属图片的信息
  2. 重点: 使用 bbox_roi_extractor 进行每个 roi 的特征信息的提取
  3. 然后将得到的 features feed 给 bbox_head (这里包含了 classification 以及 bbox regression)
  4. 最后将 loss 记录一下就可以了
  5. 然后整个 train_forward 部分就结束了, 将 losses 返回就好了 (nice!!!)

三、细节描述

PS: bbox2roi 操作就不仔细陈述了, 很简单的操作

  1. 首先我们就应该看一下 bbox_roi_extractor, 这个东西在 __init__ 中好像就已经被我们注册了, 和其他的部分一样, 都是通过 build_from_cfg 进行注册, 这个部分是使用 ROI_EXTRACTORS 这个 Registry 进行的. 同时根据 config 可以知道, 使用的是 SingleRoIExtractor, 那么就去看一下这个类: PS: 好像就只有这一个 Extractor 233
    • (1). 还是先分析一下其 __init__ 函数: __init__(self, roi_layer, out_channels, featmap_strides, finest_scale=56):
      • 1). roi_layer: 关于 RoI layer 的 config 信息
      • 2). out_channels: RoI layer 输出的通道数
      • 3). featmap_strides: 输入进来的 feature map 的 strides list
      • 4). finest_scale: mapping 到 level 0 的 threshold (??? 存疑)
      • 整个函数中基本就是参数的注册, 但是有一行要注意 -> 在里面注册了 RoI layer (… 禁止套娃!!!(滑稽)), 所以我们要关注 RoI Layer 了呢…
      • 根据配置, 我们可以发现, 使用的是 RoI Align (RoI Pooling 已经被抛弃了 2333)
      • em… RoIAlign 藏在了 mmdet.ops.roi_align.roi_align.RoIAlign… 还是从 __init__进行分析: 涉及到的四个参数: 1. out_size: 输出尺寸(config 中设置是 7) 2. spatial_scale: 空间尺度 (PS: config中没有这个参数, 是在 build_roi_layers 中根据 featmap_strides 填入的, 每个 scale 都会对应一个 RoIAlign 层) 3. sample_num: 采样点数目 4. use_torchvision: 是否使用 torchvision … (默认是 False, 话说为啥不用…). 至于 __init__ 干了什么, 就基本啥都没干, 就是进行了一下赋值操作
      • 看 forward 函数把: 参数是 features 和 rois, 如果 use_torchvision 为 True, 那么使用 torchvision.ops.roi_align, use_torchvision 为 False, 那么使用 mmdetection 定义的 roi_align. 看一下自定义的 roi_align 理清一下逻辑(应该是提供了一个 cuda 加速版本…)
      • roi_align 函数的定义是 roi_align = RoIAlignFunction.apply, 即通过自定义实现的 RoIAlignFunction 类进行的. 看一下这个类的 forward, em… 实际上是要调用 roi_align_cuda 也就是 cuda 加速版本… 我 mac 上没有 cuda 所以就安装不了这部分…
      • 看看 torchvision.ops.roi_align: 两者套路完全一致… 都是用 C 去写的核心部分 (貌似是因为涉及到了 CUDA 加速计算… 好吧 去康康, 暂时略过了… 只是大概明白了什么操作, 是要对每一个 scale 都要进行 RoIAlign 操作, 见前言的3.)
      • 这就不用再分析 ROI layer 了, 继续看 SingleRoIExtractor
    • (2). 看看 forward 部分:
      • 关于输入: feats: 就是 feature map rois: 就是 sample 出来的 RoIs 信息 roi_scale_factor: (默认值是False) 表示收对 RoI 进行 Rescale
      • 1). 首先如果 feats (也就是各个 stage 的 feature map) channel 数目只有 1(就是不使用 FPN 的 Faster RCNN) 的时候, 就直接根据仅有的 RoILayer 进行处理然后返回即可 即 return self.roi_layers[0](feats[0], rois), 但是如果使用 FPN (即 channel 数目 > 1)那么就要进行下面的操作了
      • 2). 首先会获取一些维度信息, 然后使用 map_roi_levels 对 RoI 进行 stage 的分配 (就是这里, 就是这里, 这里将不同的 RoI 按照大小分配到不同的 stage 上), 康一康 map_roi_levels:
        • map_roi_levels(self, rois, num_levels): rois 就是所有的 RoI 框. num_levels: 使用的 feautre map stage 的数目
        • 具体操作就是: 先算一下每个 proposal 的面积, 然后除以 finest_scale 之后进行一下 log2, 取一下floor, 之后 clamp 到 0 - 3(这个 config 来看是 3) 之间. 然后返回就可以
      • 3). 创建一个 roi_num * out_channel * out_size * out_size 的 Tensor 用来装最后的提取出来的 feature
      • 4). 对各个 stage 的 roi_layers 传入对应的 feature map 以及 进行了 assign 到当前 stage 的 rois 去得到提取到的 features. 得到 RoI Features, 将 RoI Features 填充到对应的地方
      • 5). 返回这个 roi_feats
  2. 后面涉及到了一个 with_shared_head 参数, 看了一下框架的代码, 只有一个 shared_head 还是 res_layer 看上去也就是这个就是用来进行额外的处理的, 就是对 提取出来的 feature map 的信息进行进一步处理之后再传递给 bbox_head. 根据配置文件可以看出, Faster RCNN 中使用的是 SharedFCBBoxHead, 查找 SharedFCBBoxHead 之后可以发现, 其有一个父类 ConvFCBBoxHead, 并且 SharedFCBBoxHead 的 __init__ 函数是直接调用父类的 __init__ 函数的, em… 也可以发现 ConvFCBBoxHead 也有父类 BBoxHead. 让我们先分析一下这两个类的 __init__ 函数吧:
    • (1). SharedFCBBoxHead.__init__: 首先看一下这个类中的注释, 描述了这个类的流程是什么
    •                                              /-> cls convs -> cls fcs -> cls
    • shared convs -> shared fcs
    •                                              \-> reg convs -> reg fcs -> reg
    • 也就是我们首先会经过 classfication 与 bbox regression 共同要使用的 conv, fc 层, 然后才拆分为两个分支, 分别进行两个任务的 conv, fc 层, 最后得到结果
    • 所以有了这个目的, 让我们看一下其中的参数吧, 但是还是最好先看一下其父类 BBoxHead 的参数:
    • (2). BBoxHead.__init__(self, with_avg_pool=False, with_cls=True, with_reg=True, roi_feat_size=7, in_channels=256, num_classes=81, target_means=[0., 0., 0., 0.], target_stds=[0.1, 0.1, 0.2, 0.2], reg_class_agnostic=False, loss_cls=dict( type='CrossEntropyLoss', use_sigmoid=False, loss_weight=1.0), loss_bbox=dict( type='SmoothL1Loss', beta=1.0, loss_weight=1.0)):
      • PS: 在 BBoxHead 中的注释说明: 这个类表示的是最简单的 RoIHead, 只有两个分别用来进行 classification 以及 bbox regression 的 FC 层.
      • 1). with_avg_pool: 表示是否在进行 FC 前使用一个尺寸大小等于输入尺寸大小的 AvgPool 层先进行一次处理.
      • 2). with_cls: 表示是否包含 classification 用途的 fc 层
      • 3). with_reg: 表示是否包含 bbox regression 用途的 fc 层
      • 4). roi_feat_size: 表示输入进来的 roi feature map 的尺寸
      • 5). in_channels: 表示输入进来的 roi feature map 的通道数
      • 6). num_classes: 表示要分类的类别数目
      • 7). target_mean && target_stds: 进行 bbox_target 时候使用的参数, 是 bbox 解码过程需要的参数, 印象中没有错的话 这部分应该是加速训练收敛的
      • 8). reg_class_agnostic: 表示是否使用类别不可知的方式进行 bbox regression (也就是 bbox regression 分支输出的就是 4, 而不是 4 * num_classes)
      • 9). loss_cls: classification 的 loss config
      • 10). loss_reg: bbox regression 的 loss config
      • PS: 我们可以看到在 Faster RCNN 的 config 文件中, bbox_head 相关参数都是给 BBoxHead.__init__ 的, 基本没有给 ConvFCBBoxHead.__init__ 的参数, 应该是都靠其默认参数了, config 中只有 num_fcs 一个参数是给 SharedFCBBoxHead 的
      • 这个 __init__ 中主要进行的就是参数在类中的注册 以及简单的注册了 fc_cls 和 fc_reg 两个 FC 层
      • 接下来看 ConvFCBBoxHead.__init__
    • (3). ConvFCBBoxHead.__init__(self, num_shared_convs=0, num_shared_fcs=0, num_cls_convs=0, num_cls_fcs=0, num_reg_convs=0, num_reg_fcs=0, conv_out_channels=256, fc_out_channels=1024, conv_cfg=None, norm_cfg=None, *args, **kwargs):
      • 1). num_shared_convs: 表示 shared_conv 层的数目
      • 2). num_shared_fcs: 表示 shared_fcs 层的数目
      • 3). num_cls_convs: 表示 cls_convs 层的数目
      • 4). num_cls_fcs: 表示 cls_fcs 层数目
      • 5). num_reg_convs: 表示 reg_convs 层的数目
      • 6). num_reg_fcs: 表示 reg_fcs 层的数目
      • 7). conv_out_channels: 表示流程中 conv 层的输出通道数
      • 8). fc_out_channels: 表示流程中 fc 层的输出维度
      • 9). conv_cfg: 关于卷积层的配置
      • 10). norm_cfg: 关于 norm 的配置 (PS: 因为 Faster RCNN 中并没有使用 conv, 无论是 shared_conv 还是分支中的 conv, 所以 norm 也是木有配置的)
      • 流程: 除了进行参数值类内的注册之外, 还是会通过 _add_conv_fc_branch 进行各个阶段(指的就是 shared_conv, shared_fc 等这一系列的流程) 然后最后会在层后面接上一个 cls_fc_output 一个 reg_fc_output em… 接下来简单分析一下 _add_conv_fc_branch 函数吧
    • (4). _add_conv_fc_branch(self, num_branch_convs, num_branch_fcs, in_channels, is_shared=False):
      • 1). num_branch_convs: 表示 conv 层的数目
      • 2). num_branch_fcs: 表示 fc 层的数目
      • 3). in_channel: 表示输入的维度
      • 4). is_shared: 表示是否是前面 shared layers 的 flag
      • 流程: 就是按照参数去决定添加一下各个层次
  3. 接下来就对 forward 函数进行分析: 没啥好分析的, 就是按照流程各个模块走一遍最后得到 cls_score 以及 bbox_pred
  4. 之后会通过 bbox_head.get_target 获取进行损失函数的 GT 信息, 这个就要追溯到 BBoxHead 这个类上了, 有函数定义: get_target(self, sampling_results, gt_bboxes, gt_labels, rcnn_train_cfg):
    • (1). sampling_results: 上一个大阶段得到的 sample 出来的 proposals 结果
    • (2). gt_bboxes: GT BBox 信息 (实际上用不到)
    • (3). gt_labels: GT Labels 信息 (实际上用不到, 这两者用不到的实际原因应该是因为在 sampling_result 中已经将这些信息包含了)
    • (4). rcnn_train_cfg: config 文件中的 train_cfg.rcnn
    • 具体流程:
    • (1). 直接从 sampling_result 中提取出来 pos_proposals, neg_proposals, pos_gt_bboxes, pos_gt_labels, 然后将这些参数直接 feed 给 bbox_target 中
    • (2). bbox_target 会通过 multi_apply 调用 bbox_target_single 对每个 stage 的信息进行分别的 target_single 处理 (因为虽然上面的 features 已经是多尺度的信息放在一起了, 但是 sampling_results 还是分尺度的, 所以要用 multi_apply)
      • 分析 bbox_target_single: 主要目的就是将 pos 和 neg 的信息整合在一起, label_weights 表示这个标签应该占据的权重, neg_weight 是 1, 如果没有在 config 中具体设置一个 >0 的权重, 那么 pos_weight 也是 1. 同时只有 pos_sample 才会使用 bbox 信息.
      • 处理之后得到 labels, label_weights, bbox_targets, bbox_weights 然后会返回给 bbox_target, 然后在其中经过汇总拼接将拼接好的多尺度的信息返回
  5. 我们已经拿到了 optimize target 了, 接下来就要去拿 loss 了, 通过 bbox_head.loss 按照配置文件进行 loss 的计算配置 然后返回 loss
  6. 我们拿到了 loss 之后, 在 two_stage.py 的 forward_train 中返回这些 loss 即可

PS: 这样我们的这部分代码也就看完了, 整个 TwoStageDetector 就阅读完毕了, 后面如果有需要的话会记录 train.py 相关的内容.
